# Identity Diagnosis: Sections 3, 6, 7

> Date: 2026-03-02
> Phase: 18.1 (Canonical View Identity System)
> Status: Diagnosed, fix strategies proposed

## Section 3: Animation Completely Broken

### Symptom
- Identity retained after idMap revert (Plan 08 rollback)
- Animation completely broken — no exit transitions on deleted cards, even for the last card
- `withAnimation { removeItem }` + `.transition(.slide)` has no effect

### Root Cause

ForEach items are wrapped with `TagModifier(.tag)` + `IdentityKeyModifier`, but the container's `idMap` closure only looks for `TagModifier(.id)`:

```swift
// VStack/HStack/ZStack idMap:
let idMap: (Renderable) -> Any? = { TagModifier.on(content: $0, role: .id)?.value }
```

Since no ForEach items have `.id()` applied (only Section 8 uses explicit `.id()`), `ids=0` for every container render → **NON-ANIMATED path always taken** → no `AnimatedContent` → no transitions ever fire.

**Log evidence**: Every single container render shows `ids=0 path=NON-ANIMATED`:
```
VStack.Render: renderables=N ids=0 path=NON-ANIMATED
HStack.Render: renderables=N ids=0 path=NON-ANIMATED
```

### Why Plan 08's `?? $0.identityKey` fix regressed

Adding `?? $0.identityKey` to the idMap caused ALL ForEach items to route through `AnimatedContent`. The problem: `AnimatedContent` creates **fresh composition scopes** for each target state transition. When the content key changes (item added/removed), all items in the new state get fresh compositions — destroying peer identity (new Store, new instanceID).

### SwiftUI vs skip-ui gap

In real SwiftUI, `withAnimation { removeItem }` works because SwiftUI's diffing system handles item removal animation automatically via ForEach's implicit Identifiable identity. No `.id()` modifier needed.

In skip-ui, transitions require the ANIMATED path (`AnimatedContent`), which requires explicit `.id()` modifiers — a fundamental mismatch.

### Fix Strategy

**Option A: Per-item `AnimatedVisibility`** (recommended)
- In the NON-ANIMATED path, track previous and current item keys
- For items that disappeared: render with `AnimatedVisibility(visible = false, exit = transition)`
- For items that appeared: render with `AnimatedVisibility(visible = true, enter = transition)`
- For items that stayed: render normally
- Preserves `key()`-based composition identity while adding transitions

**Option B: Fix AnimatedContent key preservation**
- Make AnimatedContent preserve compositions for unchanged keys across state transitions
- Would require custom AnimatedContent implementation — high complexity

**Option C: Compose `Modifier.animateItem()`**
- Available in Compose Foundation 1.7+ for lazy containers
- Doesn't help non-lazy VStack/HStack ForEach

**Key files**:
- `VStack.swift`, `HStack.swift`, `ZStack.swift` — container rendering paths
- `ForEach.swift` — identity wrapping (`.tag` vs `.id`)
- `AdditionalViewModifiers.swift` — TagModifier, `.id()` implementation

## Section 6: Identity Lost When Scrolling in List

### Symptom
- Increment top card counter, add more cards, scroll down, scroll back up
- Top card counter reset to 0, instanceID changed
- Identity completely lost when items scroll out of view

### Root Cause

`List` renders via `LazyColumn`. When items scroll off-screen, LazyColumn **disposes their compositions**. The transpiler's peer remembering uses `remember { SwiftPeerHandle(...) }` — regular `remember` is lost on composition disposal.

```kotlin
// Generated by transpiler — LOST on LazyColumn scroll-off:
val peerHandle = remember(currentHash) { SwiftPeerHandle(Swift_peer, ::Swift_retain, ::Swift_release) }

// @State uses rememberSaveable — SURVIVES scroll-off:
val rememberedFoo = rememberSaveable(stateSaver = ...) { mutableStateOf(Swift_initState_foo(peer)) }
```

When the peer is released via `SwiftPeerHandle.onForgotten()`:
- `let store = Store(initialState: ...)` — Store deallocated, state lost
- `let instanceID = UUID()` — new UUID generated on peer recreation

**Log evidence**: `produceLazyItems` shows correct UUID tags for items, confirming the lazy key system works. But the composition state (peer) within each item doesn't survive.

### SwiftUI vs Compose gap

| | SwiftUI List | Compose LazyColumn |
|---|---|---|
| View recycling | Preserves identity, reuses views | Disposes compositions entirely |
| @State survival | Always preserved | Only via `rememberSaveable` |
| let-with-default | Part of view identity | Lost on disposal (`remember`) |

### Fix Strategy

See [PeerStore Proposal](#peerstore-proposal) below.

**Key files**:
- `LazyVStack.swift` — LazyColumn rendering with `items(count:key:)`
- `KotlinBridgeToKotlinVisitor.swift` — transpiler peer remembering generation
- `ComposeStateSaver.swift` — existing `rememberSaveable` Saver infrastructure

## Section 7: CounterCard Loses Identity After Tab Switch

### Symptom
- CounterCard ("Peer Test Card") in Section 7 loses state after switching tabs
- Counter resets, instanceID changes
- PeerRememberTestView may also be affected (same mechanism)

### Root Cause

Same as Section 6. The TabView uses NavHost with:

```kotlin
navController.navigate(route) {
    popUpTo(destinationID) {
        inclusive = true
        saveState = true  // saves rememberSaveable state
    }
    launchSingleTop = true
    restoreState = true   // restores rememberSaveable state
}
```

This **disposes the tab's composition** but saves `rememberSaveable` state. Since peer remembering uses `remember` (not `rememberSaveable`), the peer is released → Store/instanceID destroyed.

The `@State var tapCount` in `PeerRememberTestView` uses `rememberSaveable` and WOULD survive, but the peer holding it is gone. On restoration, `Swift_initState_tapCount(newPeer)` returns the default value (0), and even though `rememberSaveable` restores the old saved value, the new peer's `tapCount` was already initialized to 0 before sync.

Actually — `Swift_syncState_tapCount(peer, rememberedValue)` DOES write the saved value back into the new peer. So `@State` vars should survive IF the `rememberSaveable` state is properly restored. The issue is specifically with **let-with-default** properties (Store, UUID) which are NOT @State and have no save/restore mechanism.

### Fix Strategy

See [PeerStore Proposal](#peerstore-proposal) below.

**Key files**:
- `TabView.swift` — NavHost navigation with saveState/restoreState (line ~486-498)
- `KotlinBridgeToKotlinVisitor.swift` — peer remembering generation

## PeerStore Proposal

### Problem Statement

`remember { SwiftPeerHandle(...) }` doesn't survive Compose composition disposal (LazyColumn scroll, TabView navigation). Switching to `rememberSaveable` is insufficient because:
1. `onForgotten()` releases the peer BEFORE `rememberSaveable` can restore it
2. The peer pointer (Long) becomes dangling after release
3. There's no Compose mechanism to distinguish "temporary disposal" from "permanent disposal" in `onForgotten`

### Proposed Architecture

**Parent-scoped peer cache** that outlives individual item compositions:

```
┌─────────────────────────────────────────┐
│ Parent Composition (LazyColumn parent)  │
│                                         │
│  ViewPeerStore (remember'd map)         │
│  ┌─────────────────────────────────┐    │
│  │ "uuid-A" → peer_ptr_1 (retained)│   │
│  │ "uuid-B" → peer_ptr_2 (retained)│   │
│  │ "uuid-C" → peer_ptr_3 (retained)│   │
│  └─────────────────────────────────┘    │
│                                         │
│  ┌─ LazyColumn ──────────────────────┐  │
│  │  ┌─ Item A (visible) ──────────┐ │  │
│  │  │ peer from PeerStore["uuid-A"]│ │  │
│  │  └─────────────────────────────┘ │  │
│  │  ┌─ Item B (visible) ──────────┐ │  │
│  │  │ peer from PeerStore["uuid-B"]│ │  │
│  │  └─────────────────────────────┘ │  │
│  │                                   │  │
│  │  Item C: disposed (off-screen)    │  │
│  │  → peer ALIVE in PeerStore        │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

### Implementation Steps

#### 1. skip-ui Runtime (no transpiler changes)

**New file or additions to Renderable.swift**:

```kotlin
// CompositionLocals for identity flow
val LocalViewPeerStore = staticCompositionLocalOf<ViewPeerStore?> { null }
val LocalForEachItemKey = staticCompositionLocalOf<String?> { null }

// Parent-scoped peer cache
class ViewPeerStore {
    private val cache = mutableMapOf<String, PeerEntry>()

    data class PeerEntry(val peer: Long, val retainFn: (Long) -> Unit, val releaseFn: (Long) -> Unit)

    fun get(key: String): Long? = cache[key]?.peer

    fun put(key: String, peer: Long, retainFn: (Long) -> Unit, releaseFn: (Long) -> Unit) {
        if (!cache.containsKey(key)) {
            retainFn(peer)  // Extra retain for cache
            cache[key] = PeerEntry(peer, retainFn, releaseFn)
        }
    }

    fun cleanup(activeKeys: Set<String>) {
        val removed = cache.keys - activeKeys
        for (key in removed) {
            cache.remove(key)?.let { it.releaseFn(it.peer) }
        }
    }
}
```

**IdentityKeyModifier.Render** — provide item key via CompositionLocal:

```kotlin
@Composable override func Render(content: Renderable, context: ComposeContext) {
    CompositionLocalProvider(LocalForEachItemKey provides String(describing: normalizedKey)) {
        content.Render(context: context)
    }
}
```

**LazyVStack.swift** — provide PeerStore:

```kotlin
val peerStore = remember { ViewPeerStore() }
CompositionLocalProvider(LocalViewPeerStore provides peerStore) {
    LazyColumn(...) { ... }
}
// Cleanup removed items after ForEach data change
```

**TabView.swift** — provide PeerStore (same pattern).

#### 2. Transpiler Changes (skipstone)

**Add `rememberViewPeer` helper in skip-ui runtime**:

```kotlin
@Composable
fun rememberViewPeer(
    peer: Long,
    retainFn: (Long) -> Unit,
    releaseFn: (Long) -> Unit,
    inputsHash: Long? = null
): SwiftPeerHandle {
    val store = LocalViewPeerStore.current
    val itemKey = LocalForEachItemKey.current

    if (store != null && itemKey != null) {
        val cachedPeer = store.get(itemKey)
        if (cachedPeer != null && cachedPeer != peer) {
            releaseFn(peer)  // Release stale new peer
            return remember { SwiftPeerHandle(cachedPeer, retainFn, releaseFn, managed = true) }
        } else {
            store.put(itemKey, peer, retainFn, releaseFn)
            return remember { SwiftPeerHandle(peer, retainFn, releaseFn, managed = true) }
        }
    }

    // Fallback: existing behavior
    if (inputsHash != null) {
        return remember(inputsHash) { SwiftPeerHandle(peer, retainFn, releaseFn) }
    }
    return remember { SwiftPeerHandle(peer, retainFn, releaseFn) }
}
```

**SwiftPeerHandle** — `managed` flag skips release in `onForgotten` (PeerStore handles lifecycle):

```kotlin
class SwiftPeerHandle(
    val peer: Long,
    private val retainFn: (Long) -> Unit,
    private val releaseFn: (Long) -> Unit,
    private val managed: Boolean = false
) : RememberObserver {
    init { if (!managed) retainFn(peer) }
    fun swapFrom(stale: Long) { retainFn(peer); releaseFn(stale) }
    override fun onRemembered() {}
    override fun onAbandoned() { if (!managed) releaseFn(peer) }
    override fun onForgotten() { if (!managed) releaseFn(peer) }
}
```

**Transpiler code generation change** (KotlinBridgeToKotlinVisitor.swift):

```kotlin
// Old (Phase 1):
val peerHandle = remember { SwiftPeerHandle(Swift_peer, ::Swift_retain, ::Swift_release) }

// New (Phase 1):
val peerHandle = skip.ui.rememberViewPeer(Swift_peer, ::Swift_retain, ::Swift_release)

// Old (Phase 2):
val currentHash = Swift_inputsHash(Swift_peer)
val peerHandle = remember(currentHash) { SwiftPeerHandle(Swift_peer, ::Swift_retain, ::Swift_release) }

// New (Phase 2):
val currentHash = Swift_inputsHash(Swift_peer)
val peerHandle = skip.ui.rememberViewPeer(Swift_peer, ::Swift_retain, ::Swift_release, inputsHash = currentHash)
```

#### 3. ForEach Cleanup Integration

When ForEach data changes, compute the set of active identity keys and call `peerStore.cleanup(activeKeys)`. This can be done in `ForEach.Evaluate()` or in the lazy item production callbacks.

### Scope & Risk Assessment

| Component | Files Changed | Risk |
|-----------|--------------|------|
| skip-ui runtime | Renderable.swift, LazyVStack.swift, TabView.swift, ForEach.swift | Medium — new plumbing, but isolated behind CompositionLocal null-check |
| skipstone transpiler | KotlinBridgeToKotlinVisitor.swift | Low — mechanical replacement of `remember` with `rememberViewPeer` call |
| No PeerStore fallback | All existing views | Zero — `LocalViewPeerStore.current == null` → existing `remember` behavior |

### What This Does NOT Fix

- **Section 3 animation**: Requires separate per-item `AnimatedVisibility` mechanism
- **Process death restoration**: PeerStore is in-memory only; process death still loses peers
- **Non-ForEach views in TabView**: Views not wrapped in ForEach (no identity key) won't benefit
  - Could extend with a `LocalTabKey` or position-based key for tab content
