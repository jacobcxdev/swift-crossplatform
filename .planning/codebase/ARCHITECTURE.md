# Architecture

**Analysis Date:** 2026-02-20

## Pattern Overview

**Overall:** Multi-platform Swift compilation framework (Skip) with cross-platform examples using Lite and Fuse compilation modes. The architecture uses a monorepo structure with 12 forked Point-Free/GRDB dependencies and Skip framework forks to build iOS/macOS and Android applications from unified Swift source code.

**Key Characteristics:**
- **Dual-mode compilation**: Lite (Swift transpilation to Kotlin) and Fuse (native Swift on Android with Compose bridging)
- **Observation-based state management**: Uses Swift Observation macros and TCA (The Composable Architecture) for reactive updates
- **JNI bridging**: Swift ↔ Kotlin/Compose integration for UI rendering and state synchronization
- **Submodule dependency management**: 12 forks organized under `/forks/` with synchronized branches (`flote/service-app`)
- **Skip build plugin integration**: `skipstone` generates platform-specific bridge code at build time

## Layers

**UI Layer (SwiftUI on iOS/macOS, Compose on Android):**
- Purpose: Cross-platform view definitions using SwiftUI syntax
- Location: `examples/lite-app/Sources/`, `examples/fuse-library/Sources/`
- Contains: View structs with `@Environment`, `@State`, `@Observable` bindings
- Depends on: Skip framework (skip-ui for Lite, skip-fuse for Fuse), state layer
- Used by: Application entry points (Darwin/Main.swift, Android build output)

**State Management Layer:**
- Purpose: Reactive state mutations and observation tracking
- Location: `examples/lite-app/Sources/LiteApp/ViewModel.swift`, fork packages (swift-composable-architecture, swift-perception, swift-dependencies)
- Contains: `@Observable` classes, TCA stores, registrars, persistence logic
- Depends on: Swift Observation (native or Perception backport), GRDB for persistence
- Used by: UI views via `@Environment` and direct property binding

**Bridge Layer (Skip Fuse-specific):**
- Purpose: Translate Swift Observation state changes to Compose recomposition events
- Location: `forks/skip-android-bridge/Sources/SkipAndroidBridge/Observation.swift`, generated code in `.build/plugins/outputs/*/SkipBridgeGenerated/`
- Contains: ObservationRegistrar (access/withMutation), JNI function declarations, MutableState counter management
- Depends on: Swift JNI, Compose MutableState (Kotlin side), native Swift Observation
- Used by: Generated Swift_composableBody functions, TCA Store initialization

**Build/Compilation Layer:**
- Purpose: Transpile Swift → Kotlin (Lite) or compile native Swift for Android (Fuse)
- Location: Skip build plugin (`skipstone`), `.build/` artifact directory
- Contains: Generated bridge files, transpiled Kotlin sources, Compose bindings
- Depends on: Swift compiler, Kotlin compiler, Android SDK/NDK
- Used by: Gradle, Xcode build systems

**Persistence Layer:**
- Purpose: Data serialization and SQLite storage
- Location: `forks/GRDB.swift/`, `forks/sqlite-data/`, application-level saveItems/loadItems functions
- Contains: GRDB query builders, SQLite C bindings, JSON codecs
- Depends on: SQLite library, native SQLite Kotlin bindings
- Used by: ViewModel classes for state durability

**Dependency Injection Layer:**
- Purpose: Provide shared services to views and stores
- Location: `forks/swift-dependencies/Sources/`
- Contains: DependencyKey protocol, @Dependency macro, TestDependencyKey implementations
- Depends on: Swift macros, task-local storage
- Used by: Store initialization, view environment setup

## Data Flow

**View Render → State Read (iOS):**
1. SwiftUI internally wraps body evaluation with `withObservationTracking { body() }`
2. body property accessed → reads from @Observable ViewModel
3. `access(keyPath:)` called via Observation registrar
4. Tracking scope records keypath dependency
5. SwiftUI subscribes to onChange callback
6. View renders with current state values

**State Mutation (iOS → Android):**
1. User interaction (button tap, text edit, toggle)
2. Action dispatched to TCA Store or direct @State update
3. Store reducer or property setter executes
4. For @Observable: calls `withMutation(keyPath:_:)` → mutation block executes → notifies Observation
5. Observation onChange fires → schedules single re-render
6. Next body evaluation enters withObservationTracking again

**View Render → State Read (Android Lite/Transpiled):**
1. Gradle compiles transpiled Kotlin → Java/Compose
2. Compose @Composable function renders view tree
3. View state read invokes generated Kotlin delegation
4. Kotlin calls JNI → `Swift_composableBody` (generated by skipstone)
5. Swift body property accessed → reads ViewModel state
6. Observation registrar `access()` increments counter via JNI
7. Kotlin MutableStateBacking counter read → Compose registers snapshot dependency

**State Mutation (Android Fuse/Native Swift):**
1. User action → TCA Store or direct mutation
2. Swift code executes `withMutation(keyPath:_:)` (native or Perception)
3. Observation registrar notifies both:
   - JNI bridge: increments MutableState counter → triggers Compose recomposition
   - Native Observation: fires onChange callback (unused in current implementation, candidate for fix)
4. Compose recomposes → calls Swift_composableBody again
5. **Problem (unfixed):** Every withMutation increments counter without deduplication, causing infinite loop with high-frequency mutations (TCA's _$id UUID changes)

**Data Persistence:**
1. ViewModel items change
2. `didSet` observer calls `saveItems()`
3. JSONEncoder encodes items array
4. FileManager writes to URL.applicationSupportDirectory
5. On app restart, loadItems() reads file and decodes
6. For GRDB: replace JSON with database transactions via GRDB query interface

## Key Abstractions

**@Observable (Swift Observation):**
- Purpose: Declare reactive properties with automatic keypath tracking
- Examples: `examples/lite-app/Sources/LiteApp/ViewModel.swift` line 9, `@Observable public class ViewModel`
- Pattern: Applied to classes that contain mutable `var` properties; Swift compiler generates `access()` and `withMutation()` calls

**Skip Observation Registrar:**
- Purpose: Dual-path notification: native Swift Observation + JNI to Kotlin MutableState
- Examples: `forks/skip-android-bridge/Sources/SkipAndroidBridge/Observation.swift`
- Pattern: Implements `ObservationRegistrar` protocol; `access(keyPath:)` reads counter via JNI, `withMutation(keyPath:_:)` increments counter; serves both platforms from same code

**TCA Store (The Composable Architecture):**
- Purpose: Redux-like state container with reducer functions and dependency injection
- Examples: `forks/swift-composable-architecture/Sources/ComposableArchitecture/Store.swift`
- Pattern: Generic `Store<State, Action>` initialized with initial state, reducer, and optional dependencies; synchronously mutates state, notifies observers via ObservationRegistrar

**Skip Build Plugin (skipstone):**
- Purpose: Generate platform-specific bridge code at build time
- Examples: Generated files in `.build/plugins/outputs/service-app/ServiceApp/destination/skipstone/SkipBridgeGenerated/`
- Pattern: Macro-style code generation; emits `Swift_composableBody` JNI functions, Kotlin Composable wrappers, type marshalling code

**Environment Pattern (SwiftUI):**
- Purpose: Pass shared objects (ViewModels, Services) through view hierarchy without drilling
- Examples: `examples/lite-app/Sources/LiteApp/ContentView.swift` line 38 `.environment(viewModel)` and line 62 `@Environment(ViewModel.self) var viewModel`
- Pattern: `@Environment` reads from SwiftUI environment dictionary keyed by type; @Observable classes serve as implicit keys

## Entry Points

**iOS App (Darwin):**
- Location: `examples/lite-app/Darwin/Sources/Main.swift`
- Triggers: `@UIApplicationMain` or SwiftUI App protocol initialization
- Responsibilities: Create application window, load LiteAppRootView, set up app delegate callbacks

**Android App (Compose/Generated):**
- Location: Generated by skipstone; wrapper in `.build/plugins/outputs/lite-app/LiteApp/destination/skipstone/`
- Triggers: Android Activity launch via ActivityResult
- Responsibilities: Initialize Compose runtime, call Swift_composableBody, manage Kotlin ↔ Swift object lifetime

**Example Root View:**
- Location: `examples/lite-app/Sources/LiteApp/LiteApp.swift` (LiteAppRootView struct)
- Triggers: Instantiated from App delegate
- Responsibilities: Create TabView with three NavigationStack containers, set up ViewModel environment, apply appearance theme

**ContentView (Lite Example):**
- Location: `examples/lite-app/Sources/LiteApp/ContentView.swift`
- Triggers: Loaded by LiteAppRootView
- Responsibilities: Render tab interface (Welcome, Home, Settings), manage tab state with @AppStorage, coordinate ItemListView with ViewModel

## Error Handling

**Strategy:** Minimal explicit error handling; relies on Optional unwrapping and graceful degradation.

**Patterns:**
- ViewModel persistence: `try?` and guard statements; missing/corrupted data falls back to default items (lines 67-81 in ViewModel.swift)
- File I/O: FileManager operations wrapped in `do { try ... } catch { logger.error(...) }` blocks
- Logging: OSLog logger used for warnings/errors; messages include context (file path, byte count, duration)
- View rendering: No explicit error boundaries; Skip/Compose handle malformed views by rendering as empty/placeholder

**Observation-specific error handling:**
- Registrar access/withMutation: No throw annotations; assume valid keypaths generated by compiler
- JNI calls: Assume bridge initialization succeeded; no null checks on JavaObjectPointer returns (generated code assumes success)

## Cross-Cutting Concerns

**Logging:**
- Framework: OSLog (imported in ViewModel.swift, LiteApp.swift)
- Configuration: Logger(subsystem: "dev.jacobcx.*", category: "ModuleName")
- Usage: Informational logs for data persistence timing, warning logs for load failures, error logs for I/O exceptions

**Validation:**
- Approach: Compile-time validation via Swift type system; runtime checks in ViewModel.isUpdated() to compare state
- Item equality: Codable conformance validates JSON structure; Identifiable ensures unique IDs

**Authentication/Security:**
- Approach: Not implemented in example apps; @AppStorage uses platform defaults (UserDefaults on iOS, SharedPreferences on Android via Skip)
- Observation: No explicit encryption; user data stored in application support directory with platform-level file permissions

**Platform-Specific Code:**
- Mechanism: `#if SKIP` / `#else` / `#endif` preprocessor directives
- Example: PlatformHeartView in ContentView.swift (line 160-171) renders ComposeView with Kotlin code on Android, SwiftUI Text on iOS
- Limitation: Full Swift codebase must compile in both contexts; transpiler rejects unsupported syntax (e.g., complex keypaths in Lite mode)
