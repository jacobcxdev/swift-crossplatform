  **1. ViewModifier observation hooks (OBS-06):**
  In `forks/skip-ui/Sources/SkipUI/SkipUI/View/ViewModifier.swift`, modify the `ViewModifier` extension's `Evaluate()` method (inside `#if SKIP`) to add observation recording around the body evaluation, matching the pattern in `View.Evaluate()`:
  ```swift
  @Composable public func Evaluate(content: Content, context: ComposeContext, options: Int) -> kotlin.collections.List<Renderable> {
      ViewObservation.startRecording?()

      StateTracking.pushBody()
      let renderables = body(content: content).Evaluate(context: context, options: options)
      StateTracking.popBody()

      ViewObservation.stopAndObserve?()
      return renderables
  }
  ```
  This requires importing `ViewObservation` -- it is already accessible from the same module (SkipUI). The `ViewObservation` struct is declared in `View.swift` and is module-internal.

  **2. Bridge init failure detection (OBS-04):**
  In `forks/skip-ui/Sources/SkipUI/SkipUI/View/View.swift`, the ViewObservation init block currently has `catch (_: Throwable) {}` which silently swallows errors. Per CONTEXT.md decision, bridge init failure is fatal — `fatalError()` with a clear message. Lite mode apps never call `nativeEnable()` at all (they don't depend on skip-android-bridge), so backward compat is not a concern. Change the SKIP INSERT init block to:
  ```
  / SKIP INSERT: init { try { nativeEnable(); startRecording = ::nativeStartRecording; stopAndObserve = ::nativeStopAndObserve } catch (e: Throwable) { error("ViewObservation: nativeEnable() failed. Observation bridge is NOT active. This is fatal in Fuse mode — the bridge is load-bearing infrastructure. Error: ${e.message}") } }
  ```
  Note: Kotlin's `error()` throws `IllegalStateException` which is equivalent to `fatalError()` — it crashes the app immediately with a clear message. This is the correct behavior per CONTEXT.md: "Bridge initialization failure is fatal. If `ViewObservation.nativeEnable()` fails or JNI exports don't resolve, `fatalError()` with a clear message."

  **3. Per-call JNI failure is fatal (CONTEXT.md decision):**
  In `forks/skip-ui/Sources/SkipUI/SkipUI/View/View.swift`, the `startRecording` and `stopAndObserve` closures are currently assigned directly as `::nativeStartRecording` and `::nativeStopAndObserve` in the init block. Per CONTEXT.md: "Runtime JNI failures are fatal per call. If `nativeStartRecording()` or `nativeStopAndObserve()` fails mid-session, `fatalError()` immediately."

  Wrap the closure assignments so that any JNI call failure at runtime is fatal. In the init block (after the successful `nativeEnable()` call), assign:
  ```kotlin
  startRecording = {
      try { nativeStartRecording() }
      catch (e: Throwable) { error("ViewObservation: nativeStartRecording() failed mid-session. JNI bridge is broken. Error: ${e.message}") }
  }
  stopAndObserve = {
      try { nativeStopAndObserve() }
      catch (e: Throwable) { error("ViewObservation: nativeStopAndObserve() failed mid-session. JNI bridge is broken. Error: ${e.message}") }
  }
  ```
  This ensures any runtime JNI resolution or invocation failure crashes immediately with a clear diagnostic, rather than silently producing broken observation behavior.

  **5. onChange main thread dispatch (context decision):**
  In `forks/skip-android-bridge/Sources/SkipAndroidBridge/Observation.swift`, modify `ObservationRecording.stopAndObserve()` to dispatch the trigger closure to main when onChange fires from an arbitrary thread. Replace the current onChange closure in `stopAndObserve()`:
  ```swift
  ObservationModule.withObservationTrackingFunc({
      for closure in closures {
          closure()
      }
  }, onChange: {
      DispatchQueue.main.async {
          trigger()
      }
  })
  ```
  The `trigger()` call invokes `BridgeObservationSupport.triggerSingleUpdate()` which does `Java_update(0)` -- a JNI call to `MutableStateBacking.update()`. Since Compose state mutations should happen on the main/UI thread, dispatching to main ensures correctness. The `DispatchSemaphore` inside `triggerSingleUpdate()` handles reentrancy if `main.async` coalesces multiple calls.

  All methods follow the existing pattern: static function returning Bool, using `withObservationTracking`, `@Observable` models, and `FlagBox` for onChange detection.

  1. **`verifyBulkMutationCoalescing()`** (OBS-15): Mutate multiple properties of the same `@Observable` object in sequence (e.g., `counter.count = 1; counter.count = 2; counter.count = 3`). Verify onChange fires (at least once). This validates that bulk mutations don't cause N separate recompositions. Note: native `withObservationTracking`'s onChange fires once then auto-cancels, so multiple mutations after the first onChange has fired will not trigger again until re-subscription. This is the correct behavior.

  2. **`verifyObservationIgnoredNoTracking()`** (OBS-17, more thorough): Create a `Counter`, track `ignoredValue` in `withObservationTracking`, mutate a NON-ignored property (`count`), verify that `count` mutation does NOT fire onChange (because `ignoredValue` was the only accessed property and it's ignored, so no tracking was set up). This tests that `@ObservationIgnored` truly suppresses tracking even for the observation registrar.

  3. **`verifyViewModifierIndependentFrame()`** (OBS-06 bridge-level): Simulate the ViewModifier frame independence pattern. Call `ObservationRecording.startRecording()` (outer view), then `ObservationRecording.startRecording()` (inner modifier), then `ObservationRecording.stopAndObserve()` (inner), then `ObservationRecording.stopAndObserve()` (outer). Verify both complete without error and the frame stack is empty afterward. This requires `ObservationRecording` to be importable -- add `import SkipAndroidBridge` with `#if canImport(SkipAndroidBridge)` guard, and wrap the test in a platform check. On macOS, this test verifies the logic by directly calling the recording API. On Android, the actual JNI path is tested via `skip test`.

  Note: `ObservationRecording` is in `SkipAndroidBridge` which is only available when `SKIP_BRIDGE` is defined. For macOS unit tests, we cannot import it directly. Instead, create a simplified test that verifies the frame stack concept using native `withObservationTracking` nesting:
  ```swift
  /// Verifies nested observation tracking (simulating parent view + child modifier).
  public static func verifyNestedObservationCycles() -> Bool {
      let parent = Counter()
      let child = Counter()
      let parentFlag = FlagBox()
      let childFlag = FlagBox()

      // Outer tracking (parent view)
      withObservationTracking {
          _ = parent.count
          // Inner tracking (child modifier)
          withObservationTracking {
              _ = child.count
          } onChange: {
              childFlag.value = true
          }
      } onChange: {
          parentFlag.value = true
      }

      // Mutate child -- should fire child's onChange
      child.count = 1
      let childFired = childFlag.value

      // Mutate parent -- should fire parent's onChange
      parent.count = 1
      return childFired && parentFlag.value
  }
  ```

  4. **`verifySequentialObservationCyclesResubscribe()`** (OBS-01 cycle behavior): Set up observation, trigger onChange, then set up a NEW observation on the same object. Verify the second cycle also fires onChange. This validates the re-subscription pattern that Evaluate() relies on (each recomposition sets up a fresh observation).

  5. **`verifyMultiPropertySingleOnChange()`** (OBS-22 batched access): Track multiple properties from different `@Observable` objects in a single `withObservationTracking` call. Mutate one property. Verify onChange fires exactly once. This validates that multiple `access()` calls in one tracking scope produce a single onChange trigger.
