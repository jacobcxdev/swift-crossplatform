Create `StructuredQueriesTests.swift` with tests covering all 15 SQL requirements. Use an in-memory `DatabaseQueue()` for all tests.

Define a `@Table` test model at file scope (macros require file-scope types per Phase 3 decision):

```swift
import StructuredQueries
import SQLiteData
import GRDB

@Table
struct Item: Identifiable, Codable, Sendable, Equatable {
    @Column(primaryKey: true)
    var id: Int
    var name: String
    var value: Int = 0
    var isActive: Bool = true
}
```

Also define a second `@Table` model for join tests:

```swift
@Table
struct Category: Identifiable, Codable, Sendable, Equatable {
    @Column(primaryKey: true)
    var id: Int
    var name: String
}
```

And a `@Selection` type for multi-column grouping (SQL-04):

```swift
@Selection
struct ItemSummary {
    var isActive: Bool
    @Column(as: QueryCount<Item>)
    var count: Int
}
```

Write a shared helper `setupSchema(_ db: DatabaseQueue)` that creates both tables via raw SQL.

Tests to write (one per SQL requirement):

| Test | Requirement | Validates |
|------|------------|-----------|
| `testTableMacro` | SQL-01 | `@Table` generates metadata — `Item.tableName`, column accessors exist |
| `testColumnPrimaryKey` | SQL-02 | `@Column(primaryKey:)` — insert without explicit id, auto-increment works |
| `testColumnCustomRepresentation` | SQL-03 | `@Column(as:)` — custom column representation via `@Selection` |
| `testSelectionTypeComposition` | SQL-04 | `@Selection` multi-column grouping with `ItemSummary` |
| `testSelectColumns` | SQL-05 | `Item.select { ($0.name, $0.value) }` returns tuple of selected columns |
| `testWherePredicates` | SQL-06 | `Item.where { $0.value > 10 && $0.isActive == true }` filters correctly |
| `testFindById` | SQL-07 | `Item.find(1)` returns the row with id=1 |
| `testWhereInOperator` | SQL-08 | `Item.where { $0.name.in(["alpha", "gamma"]) }` returns matching rows |
| `testJoinOperations` | SQL-09 | `Item.join(Category.self, on: ...)` — validate join produces combined results |
| `testOrderBy` | SQL-10 | `Item.order { $0.name.asc }` and `.desc` produce correct ordering |
| `testGroupByAggregation` | SQL-11 | `Item.group { $0.isActive }` with `count()` aggregation |
| `testLimitOffset` | SQL-12 | `Item.limit(2, offset: 1)` returns correct page |
| `testInsertAndUpsert` | SQL-13 | `Item.insert { }` creates row; `Item.upsert { }` with conflict resolution |
| `testUpdateAndDelete` | SQL-14 | `Item.where(...).update { }` mutates; `Item.find(id).delete()` removes |
| `testSqlMacro` | SQL-15 | `#sql("SELECT * FROM items WHERE value > \(Item.Columns.value, >: 10)")` or equivalent safe interpolation |

For each test:
1. Set up in-memory `DatabaseQueue()`
2. Call `setupSchema(db)`
3. Insert seed data via `db.write { try Item.insert { ... }.execute($0) }`
4. Execute the StructuredQueries operation under test
5. Assert results with `XCTAssertEqual`

**Important:** The exact StructuredQueries API may differ slightly from the examples in research. If a specific API (like `#sql()` or `@Selection`) has a different signature, adapt the test to match the actual API. Consult the fork source in `forks/swift-structured-queries/Sources/` for accurate signatures.

**Join test note:** If `Item.join(Category.self, ...)` requires a foreign key column on Item, add a `categoryId: Int?` column to the Item model and update the schema accordingly. Alternatively, if the join API uses a different pattern, match the fork's actual API.
