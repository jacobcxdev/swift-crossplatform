**File 1: ObservableState.swift (line 9)**

Change the guard from `#if !os(visionOS) && !os(Android)` to `#if !os(visionOS)`:

```swift
#if !os(visionOS)
  public protocol ObservableState: Perceptible {
    var _$id: ObservableStateID { get }
    mutating func _$willModify()
  }
#else
  public protocol ObservableState: Observable {
    var _$id: ObservableStateID { get }
    mutating func _$willModify()
  }
#endif
```

This makes `ObservableState` inherit from `Perceptible` on Android (as it does on iOS). On Android, `Perceptible` is available because swift-perception's core types compile on all platforms. The `Observable` fallback is only needed on visionOS (which has a different observation model).

**Important:** On Android, Store already conforms to both `Observable` (via `Store: Observable` in Store.swift line 629 `#if canImport(Observation)`) and will now also conform to `Perceptible` (transitively through ObservableState or directly). The `@_disfavoredOverload` on Perceptible methods ensures the Observable overloads are preferred, avoiding ambiguity.

**File 2: ObservationStateRegistrar.swift**

**Change 1 (line 8):** The registrar property selection. Currently:
```swift
#if !os(visionOS) && !os(Android)
  let registrar = PerceptionRegistrar()
#elseif os(Android)
  let registrar = SkipAndroidBridge.BridgeObservation.BridgeObservationRegistrar()
#else
  let registrar = Observation.ObservationRegistrar()
#endif
```

**Keep this unchanged.** On Android, the registrar MUST remain `BridgeObservationRegistrar` — this is the correct observation path for JNI-based Compose recomposition. The Perceptible-constrained methods will call through to `self.registrar` which is a `BridgeObservationRegistrar` on Android. Since `BridgeObservationRegistrar` has `access`, `withMutation`, etc. methods that accept `Observable`-constrained subjects, and since on Android types conform to `Observable`, the calls will work correctly through the bridge path.

**Change 2 (line 131):** The Perceptible-constrained method extension. Change from `#if !os(visionOS) && !os(Android)` to `#if !os(visionOS)`:

```swift
#if !os(visionOS)
  extension ObservationStateRegistrar {
    @_disfavoredOverload
    @inlinable
    public func access<Subject: Perceptible, Member>(
    // ... rest of methods unchanged
  }
#endif
```

This enables the Perceptible-constrained overloads on Android. These are `@_disfavoredOverload` so the Observable-constrained overloads (from the `#if canImport(Observation)` block at lines 29-129) will be preferred when the subject conforms to Observable. This is correct — on Android, the Observable path through BridgeObservationRegistrar is the real observation mechanism.

**However**, there is a compilation concern: on Android, `self.registrar` is `BridgeObservationRegistrar`, but the Perceptible methods call `self.registrar.access(subject, keyPath:)` where subject is `Perceptible` not `Observable`. BridgeObservationRegistrar's methods are constrained on `Observable`. This means the Perceptible overloads would fail to compile on Android because BridgeObservationRegistrar doesn't accept Perceptible subjects.

**Solution:** The Perceptible methods call `self.registrar.access(...)` etc. On non-Android, `self.registrar` is `PerceptionRegistrar` which has Perceptible-constrained overloads. On Android, `self.registrar` is `BridgeObservationRegistrar` which only has Observable-constrained methods.

To resolve this, we need to keep the guard as-is for the REGISTRAR PROPERTY (so Android uses BridgeObservationRegistrar), but the Perceptible extension needs to be guarded differently. Two approaches:

**Approach A (preferred):** Keep `#if !os(visionOS) && !os(Android)` on line 131 as-is. The Perceptible methods are only callable when registrar is PerceptionRegistrar. On Android, TCA code uses the Observable overloads (which go through BridgeObservationRegistrar). Since `ObservableState: Perceptible` AND types conform to `Observable`, the Observable overloads (non-disfavored) win.

**Approach B:** Enable the Perceptible extension on Android but change the method bodies to call through BridgeObservationRegistrar using an `as? Observable` cast. Adds complexity for no benefit.

**Decision: Use Approach A.** Keep `#if !os(visionOS) && !os(Android)` on line 131. The Perceptible-constrained methods are API surface for PerceptionRegistrar-backed registrars. On Android with BridgeObservationRegistrar, TCA uses the Observable overloads. The protocol inheritance change (`ObservableState: Perceptible`) is the important part — it ensures macro-generated conformances resolve correctly.
