**UIPatternTests.swift:** Write tests proving SwiftUI patterns compile and TCA data flows work correctly. Use `import Testing` and `@Test` functions. Per D6: tests BOTH compile SwiftUI patterns (proving APIs exist in Skip) AND validate data flow through Store.

**Reducers to define (file-scope):**

1. `AsyncFeature` ŌĆö Async work in effects:
   - `@ObservableState struct State: Equatable { var loadingComplete = false; var count = 0 }`
   - `enum Action { case startLoading, loadingFinished, increment }`
   - `.startLoading` returns `.run { send in await send(.loadingFinished) }` (simulates async work)
   - `.loadingFinished` sets `state.loadingComplete = true`
   - `.increment` increments count

2. `BindingExtFeature` ŌĆö Custom binding extensions:
   - `@ObservableState struct State: Equatable { var text = ""; var isOn = false }`
   - `BindableAction` conformance + `BindingReducer()`
   - Uses `$store.text` and `$store.isOn` binding projections

3. `NestedObservableFeature` ŌĆö Nested observable:
   - `@ObservableState struct State: Equatable { var parent = ParentModel() }`
   - `struct ParentModel: Equatable { var child = ChildModel(); var name = "" }`
   - `struct ChildModel: Equatable { var value = 0 }`
   - Tests that mutating `state.parent.child.value` propagates correctly through nested struct observation

4. `FormFeature` ŌĆö Multiple buttons in form:
   - `@ObservableState struct State: Equatable { var buttonATapped = false; var buttonBTapped = false; var buttonCTapped = false }`
   - `enum Action { case tapA, tapB, tapC }`
   - Each action sets its respective flag

5. `SheetToggleFeature` ŌĆö Simple isPresented sheet:
   - `@ObservableState struct State: Equatable { var showSheet = false; var sheetCount = 0 }`
   - `enum Action { case toggleSheet, incrementInSheet }`
   - `.toggleSheet` flips `showSheet`; `.incrementInSheet` increments `sheetCount`

**Tests to write:**

Task / async (UI-01):
- `testAsyncTaskInActionClosure` ŌĆö Store<AsyncFeature>: send `.startLoading`, verify `.loadingFinished` received and `loadingComplete == true`. This proves `Task { await method() }` pattern works via Effect.run.
- `testMultipleAsyncEffects` ŌĆö Send `.startLoading` twice concurrently, verify both complete (no deadlock)

Custom Binding extensions (UI-02):
- `testDynamicMemberLookupBinding` ŌĆö Store<BindingExtFeature>: verify `store.text` is readable, `store.text = "hello"` writes through binding projection. This validates dynamic member lookup binding derivation.
- `testBindingProjectionChain` ŌĆö Verify `$store.text` produces a valid `Binding<String>` type at compile time and reads/writes correctly

@State tracking (UI-03):
- `testStateInitialization` ŌĆö Store<AsyncFeature> initialized with `count: 5`, verify `store.count == 5`. Validates @ObservableState (Skip's @State equivalent for TCA) initialization.

State mutation triggers re-evaluation (UI-04):
- `testStateMutationSingleUpdate` ŌĆö Store<AsyncFeature>: send `.increment` 3 times, verify `store.count == 3`. Each mutation routes through reducer correctly (single update per action, not batched or duplicated).

.sheet(isPresented:) (UI-05):
- `testSheetIsPresentedToggle` ŌĆö Store<SheetToggleFeature>: send `.toggleSheet`, verify `showSheet == true`; send `.toggleSheet` again, verify `showSheet == false`. Validates binding-driven sheet lifecycle.
- `testSheetContentInteraction` ŌĆö Toggle sheet on, send `.incrementInSheet`, verify `sheetCount == 1`, toggle sheet off and verify state persists.

.task modifier (UI-06):
- `testTaskModifierPattern` ŌĆö Compile-time validation: create a view struct that uses `.task { await store.send(.startLoading).finish() }` pattern. Test that the Store + Effect.run pattern correctly simulates `.task` lifecycle (send on appear, cancel on disappear via task cancellation).

Nested @Observable (UI-07):
- `testNestedObservableGraphMutation` ŌĆö Store<NestedObservableFeature>: mutate deeply nested `state.parent.child.value = 42`, verify state propagates. Mutate `state.parent.name = "test"`, verify sibling unchanged. This validates observation tracking through nested struct hierarchy.
- `testNestedObservableIndependence` ŌĆö Verify mutating child does not disturb parent-level sibling properties (observation granularity)

Multiple buttons in Form (UI-08):
- `testFormMultipleButtonsIndependent` ŌĆö Store<FormFeature>: send `.tapA`, verify only `buttonATapped == true` and others false. Send `.tapB`, verify both A and B true, C false. Send `.tapC`, verify all true. Proves each button triggers independent action.

**Testing philosophy (per D6):** These tests validate data flow through TCA Store, which is the Android execution path. View-level semantics ("exactly one recomposition", "fires on appearance") require a running view hierarchy and are deferred to Phase 7 integration testing.
