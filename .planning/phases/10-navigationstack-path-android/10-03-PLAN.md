**Audit Methodology (execute each step):**

**Step A: skip-ui fork diff audit.**
1. Run `git diff main..dev/swift-crossplatform --name-only` in `forks/skip-ui/` to get all modified files.
2. For each modified file, check whether the modification has a corresponding counterpart in `forks/skip-fuse-ui/Sources/SkipSwiftUI/`.
3. Assess: does the skip-fuse-ui counterpart need updating to reflect the skip-ui change? Or does the skip-ui change only affect the Kotlin/Compose layer (no Swift wrapper needed)?

**Step B: skip-fuse-ui upstream comparison.**
1. Run `git diff main..dev/swift-crossplatform --name-only` in `forks/skip-fuse-ui/` to identify our additions vs upstream.
2. Check `git status` for uncommitted changes (research flagged this risk — the ModifiedContent fix may be uncommitted).
3. Document what we've added, what upstream provides, and what's still missing.

**Step C: `#if !SKIP_BRIDGE` file assessment.**
The research mentions ~148 files in skip-ui with `#if !SKIP_BRIDGE` guards. These are files that provide APIs when SKIP_BRIDGE is NOT defined (i.e., in Fuse mode they're excluded, and skip-fuse-ui's SkipSwiftUI provides the Fuse-mode equivalent).
1. Identify which of these provide APIs that TCA, fuse-app, or fuse-library actually import/use.
2. Check if skip-fuse-ui's 135 SkipSwiftUI files cover all needed APIs.
3. Document missing APIs as gaps.

**Step D: TCA cross-fork guard audit.**
1. Grep for `#if !os(Android)` and `#if canImport(SwiftUI) && !os(Android)` across `forks/swift-composable-architecture/Sources/`.
2. For each guard location, assess:
   - Is the guarded code using APIs unavailable on Android? (correct guard)
   - Is the guarded code using APIs that skip-fuse-ui now provides? (potential incorrect guard — candidate for removal)
   - Does the guarded code have an `#else` Android fallback? Is the fallback correct?
3. Special attention to NavigationStack+Observation.swift line 150 guard — research says skip-fuse-ui's generic NavigationStack MAY allow the extension to compile directly, potentially making the 10-01 adapter redundant. Test this explicitly.

**Step E: swift-navigation and swift-sharing guard audit.**
1. Grep for `#if !os(Android)` in `forks/swift-navigation/Sources/` and `forks/swift-sharing/Sources/`.
2. Assess each guard for correctness with skip-fuse-ui layer.

**Step F: Dismiss mechanism investigation.**
1. Trace the full dismiss chain from CONTEXT.md research (PresentationReducer -> DismissEffect -> Dismiss.swift -> EnvironmentValues).
2. Verify Dismiss.swift lines 90 and 122 guards are correct (withTransaction is fatalError on Android, so the guards SHOULD stay).
3. Check whether the Android fallback path (lines 98-119) correctly calls the PresentationReducer-provided dismiss closure.
4. Find the `withKnownIssue` wrappers in integration tests and assess whether dismiss should actually work.
5. Verdict: "works", "partially works", or "broken" with specific reasoning.

**Step G: JVM type erasure investigation.**
1. Examine how skip-fuse-ui's Navigation.swift handles `navigationDestination(for:)` type matching.
2. Check what `String(describing: type(of: value))` returns for `StackState<State>.Component` on JVM.
3. Assess whether single-destination-type apps (like ContactsFeature with one Component type) are safe while multi-destination apps may have issues.
4. Verdict: "safe", "risk for multi-destination", or "broken" with recommended mitigation.

**Step H: BridgeSupport and Fuse-only type audit.**
Per CONTEXT.md: include StateSupport.swift, EnvironmentSupport.swift, ComposeView, composeModifier, JavaBackedView.
1. Verify skip-android-bridge <-> skip-fuse-ui integration points.
2. Check that our skip-ui changes don't break Fuse-only types.

**Step I: New dependency edge evaluation.**
Per CONTEXT.md: check if any fork that touches UI/observation on Android needs skip-fuse-ui as a dependency but currently lacks it.
1. Check swift-composable-architecture, swift-navigation, swift-sharing, swift-perception dependency declarations.
2. Verify no cycles would be introduced.
