**TestStoreTests.swift — write tests covering:**

1. **TEST-01 — TestStore init:** `TestStore(initialState: Feature.State()) { Feature() }` initializes with correct state. Assert initial state matches.

2. **TEST-02 — send with state assertion:** `await store.send(.increment) { $0.count = 1 }`. Use a simple counter reducer. Assert state mutation in trailing closure.

3. **TEST-03 — receive effect action:** Reducer that returns `Effect.run { send in await send(.response("ok")) }`. After `store.send(.fetch)`, call `await store.receive(\.response) { $0.result = "ok" }`.

4. **TEST-04 — exhaustivity on (default):** Create TestStore, send action that changes TWO state properties, only assert ONE in trailing closure. The test should detect the unasserted change. Wrap in `withKnownIssue` to capture the expected failure (TCA reports unasserted state changes as test failures).

5. **TEST-05 — exhaustivity off:** Set `store.exhaustivity = .off`. Send action that changes state. Omit state assertion. Test should pass without failure.

6. **TEST-06 — finish():** Reducer returns a long-running effect. Call `await store.send(.start)`, then `await store.finish()`. Effect must complete. Verify final state reflects effect completion. Set `store.timeout = 5_000_000_000` for emulator safety.

7. **TEST-07 — skipReceivedActions():** Reducer returns effect sending multiple actions. Call `store.send(.triggerMultiple)`, then `await store.skipReceivedActions()`. Test passes without asserting each received action.

8. **TEST-09 — .dependencies trait:** Use `withDependencies { $0.uuid = .incrementing }` in TestStore init. Reducer uses `@Dependency(\.uuid)`. Assert generated UUID matches expected sequential value.

**Test reducer pattern:** Define inline `@Reducer` structs inside the test file (established pattern from Phase 3). Each test gets a minimal reducer exercising exactly the API under test.

**Android compatibility notes:**
- Do NOT use `store.$binding` syntax (excluded on Android per R1b §Guard 5)
- Do NOT assert concurrent effect ordering (use effect SET not SEQUENCE per R1b §Guard 4)
- Set `store.timeout = 5_000_000_000` on any test that calls `finish()` (emulator safety per R1b §Gap 3)

**TestStoreEdgeCaseTests.swift — write tests for the 4 confirmed coverage gaps from research (R1b):**

1. **Chained effects (TEST-08 gap 1):** Effect A returns action that triggers Effect B. Verify both effects settle and all intermediate state changes are assertable. This tests the `effectDidSubscribe` yield path for chained effect → action → effect sequences.

```swift
@Reducer struct ChainFeature {
    struct State: Equatable { var step = 0 }
    enum Action { case startChain, chainStepA, chainStepB }
    var body: some ReducerOf<Self> {
        Reduce { state, action in
            switch action {
            case .startChain:
                return .run { send in await send(.chainStepA) }
            case .chainStepA:
                state.step = 1
                return .run { send in await send(.chainStepB) }
            case .chainStepB:
                state.step = 2
                return .none
            }
        }
    }
}
```

2. **cancelInFlight rapid re-send (TEST-08 gap 2):** Send `.fetch` with `cancelInFlight: true`, immediately send `.fetch` again. Assert only the second effect's response arrives. Tests the brief zero-effect window between cancel and re-subscribe.

3. **finish() with long-running effect (TEST-08 gap 3):** Effect that uses `Task.sleep(for: .milliseconds(500))`. Call `await store.send(.start)`, then `await store.finish()`. Set `store.timeout = 5_000_000_000`. Verify the effect completes and state is updated.

4. **Non-exhaustive receive with .off (TEST-08 gap 4):** Set `store.exhaustivity = .off`. Reducer returns effect sending action. Call `store.send(.trigger)` without calling `receive`. Test should pass because exhaustivity is off.

**Android compatibility:** These tests validate the `effectDidSubscribe` AsyncStream fallback path that Android uses instead of `useMainSerialExecutor`. They must NOT assert on concurrent effect ordering — only that all effects eventually complete.
