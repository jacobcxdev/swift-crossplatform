<objective>
Migrate lazy container identity key generation from `composeBundleString()` to `normalizeKey()` via a lazy-key adapter function. This unifies the identity system so that both eager and lazy containers use the same normalisation pipeline, completing the canonical view identity system.

Purpose: Phase 18.1 is "implement canonical view identity system." Having eager containers on `normalizeKey()` and lazy containers on a separate `composeBundleString` mechanism is not canonical. This plan bridges the gap with a thin adapter that preserves the `(Int) -> String` type signature required by Compose's lazy `items()` API while using `normalizeKey()` internally.

**Critical constraint:** `Navigation.swift` uses `composeBundleString` for route serialisation, NOT identity. It stays as-is (explicitly N/A per recommendation doc section "Files Modified"). This plan does NOT deprecate `composeBundleString()` — it remains available for navigation use.

Output: All 6 lazy container files migrated. `composeBundleNormalizedKey()` adapter function added to `ComposeStateSaver.swift`. Equivalence tests verify key stability.
</objective>

**Step 1: Add the lazy-key adapter function to ComposeStateSaver.swift:**

Add `composeBundleNormalizedKey()` alongside the existing `composeBundleString()`:

```swift
/// Lazy-key adapter: wraps normalizeKey() to produce String keys for Compose's lazy items() API.
///
/// Compose's LazyColumn/LazyRow/LazyGrid `items(count:key:)` requires `(Int) -> String` key closures.
/// This function routes through normalizeKey() for canonical normalisation, then converts to String.
/// The result is equivalent to composeBundleString() for all common types (String, Int, UUID, enums)
/// but uses the canonical normalisation pipeline instead of ad-hoc String(describing:) conversion.
///
/// - Note: Navigation.swift uses composeBundleString() for route serialisation — that is NOT identity
///   and stays as-is.
func composeBundleNormalizedKey(for value: Any?) -> String {
    guard let value else { return "nil" }
    let normalized = normalizeKey(value)
    if let str = normalized as? String {
        return str
    }
    return String(describing: normalized)
}
```

Place this directly below the existing `composeBundleString()` function.

**Step 2: Write equivalence tests BEFORE migrating any lazy container.**

The key concern is scroll-position stability: if the migrated function produces different key strings for the same data, lazy containers will lose their scroll position on recomposition.

Write a set of assertions (in a test file or as inline verification) that confirm `composeBundleNormalizedKey(for:)` produces the **same string** as `composeBundleString(for:)` for all common identifier types:

Test cases (these are the types ForEach identifiers produce):
1. `String` value: `composeBundleNormalizedKey(for: "hello")` == `composeBundleString(for: "hello")` == `"hello"`
2. `Int` value: `composeBundleNormalizedKey(for: 42)` == `composeBundleString(for: 42)` == `"42"`
3. `UUID` value: for any UUID, both functions produce the same string (UUID is not Identifiable, not RawRepresentable — both hit String(describing:) fallback)
4. `Identifiable` struct with String id: both extract `.id` then produce same string
5. `Identifiable` struct with UUID id: both extract `.id` then produce same string via String(describing:)
6. `Identifiable` struct with Int id: both extract `.id` then produce same string

8. `RawRepresentable` enum with Int rawValue: both extract `.rawValue` then produce same string
