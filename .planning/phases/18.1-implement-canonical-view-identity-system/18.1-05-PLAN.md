**IMPORTANT: Consult `/pfw-composable-architecture` before writing ANY TCA code.** Verify patterns for: TestStore usage, @ViewAction, IdentifiedArray mutations, Binding in TCA, @Shared reactivity.

Refactor `IdentityFeature.swift` to include ALL of the following identity scenarios. Organize the view as a sectioned ScrollView with clear headers for each scenario. The existing ForEach-in-VStack card deletion scenario is retained and enhanced.

**Section 1: Eager Container Keying (VStack/HStack/ZStack) — Plans 01, 03**

Retain and enhance the existing card deletion scenario:
- `ForEach(store.cards)` in a `VStack` — already exists
- Add: "Reorder" functionality (move last card to first position) to test identity stability during reorder
- Add: Inline display of each card's `instanceID` prefix AND `store.count` — makes state preservation visually obvious
- **Manual test:** Increment counters on cards, delete middle card → remaining cards keep their counter values and instanceIDs
- **Manual test:** Increment counters, reorder → cards keep state after position change

Add a parallel HStack section (horizontal scroll) with a smaller set of items:
- Same ForEach-over-IdentifiedArray pattern but in HStack layout
- Demonstrates identity preservation works in horizontal containers too

Add a ZStack section:
- ForEach items rendered as overlapping cards (offset by index)
- Delete an item → remaining items retain state (ZStack previously had NO key() wrapping — Plan 01 Phase 0 fix)

**Section 2: Duplicate Key Guard — Plan 01**

`IdentifiedArrayOf<CardItem>` enforces unique IDs, so duplicate keys CANNOT be tested at the TCA state level. Instead, test the `seenKeys` duplicate guard at the rendering level:
- Add a NON-TCA subsection that renders a raw `[String]` array with intentional duplicates (e.g. `["Alpha", "Beta", "Alpha", "Gamma"]`) through `ForEach` using `\.self` as the id keypath — this produces duplicate keys ("Alpha" appears twice)
- Each item renders as a small counter card (local `@State var count = 0`) so state independence is observable
- Display a label showing "Duplicate keys present" to confirm the scenario is active
- **Manual test (Android):** Increment the counter on the first "Alpha" card. Verify the second "Alpha" card's counter is independent (not shared). Verify no crash occurs.
- **NOTE:** This section has NO TCA TestStore test — `IdentifiedArray` prevents the scenario. The duplicate-key `_dup` suffix guard is a Compose-level behavior verified via Android integration test or manual emulator testing only.

**Section 3: Animated Content — Plan 03**

Add an animated deletion section:
- Toggle `withAnimation` for deletions (button or switch)
- When animated: deleted card should visually animate out (fade/slide)
- Remaining cards should retain their counter state through the animation
- **Manual test:** Enable animation → increment counters → delete middle card → watch exit animation → verify remaining counters unchanged

**Section 4: Picker Selection Correctness — Plan 01**

Add a Picker subsection with multiple styles:
- **Segmented Picker:** `Picker("Style", selection: $store.selectedStyle) { ForEach(styles) { Text($0.name).tag($0) } }` with `.pickerStyle(.segmented)`
- **Menu Picker:** Same data with `.pickerStyle(.menu)`
- Display the currently selected value below each picker
- **Manual test:** Tap each segment/menu item → displayed selection matches tapped item
- This verifies that `selectionTag` works correctly after ForEach identity refactoring

**Section 5: TabView Selection Correctness — Plan 01**

Add a mini TabView subsection (nested TabView with 3 tabs):
- Each tab shows its name and a counter
- Tab selection binding displayed as text
- **Manual test:** Tap each tab → selection binding updates correctly → tab content matches
- This verifies TabView selection works after tag decoupling

**Section 6: Lazy Container Identity — Plan 04**

Add a LazyVStack section:
- `List` or `LazyVStack` with ForEach over a large-ish dataset (20+ items)
- Each item has a counter (local @State or local Store)
- Scroll down, increment some counters, scroll back up → counters retained
- Add/delete items → scroll position stable, counters on remaining items retained
- **Manual test:** Scroll to item 15, increment its counter, scroll to top, scroll back to item 15 → counter value preserved

**Section 7: Transpiler Peer Remembering — Plan 02**

The existing `CounterCard` exercises mixed views (let-with-default `store` + let-with-default `instanceID`), but Plan 02's bug is specifically about views with BOTH `@State` properties AND `let`-with-default parameters triggering the `stateVariables.isEmpty` guard in the transpiler. Add a purpose-built test view:

- Create `PeerRememberTestView` with:
  - `@State var tapCount: Int = 0` (a real @State property — NOT a Store)
  - `let color: Color = .blue` (a let-with-default parameter)
  - NO constructor params (isolates the transpiler guard behavior)
  - Body: a colored button that increments `tapCount`, displaying current count
- Add a label explicitly showing "Test: @State + let-with-default (no constructor params)"
- The existing `CounterCard` is retained as a SEPARATE subsection showing the "mixed view with Store" pattern
- **Manual test (Android):** Tap the `PeerRememberTestView` button 3 times. Then trigger parent recomposition (e.g., add a new card above). Verify `tapCount` is still 3 — without peer remembering, `@State` resets to 0 every recomposition on Android.
- **NOTE:** This section is verified via Android integration test ONLY — TCA TestStore cannot test transpiler/Compose peer remembering behavior. No TestStore test is created for this section.

**Section 8: .id() State Reset — Plan 01, 03**

Add an `.id()` reset demonstration:
- A counter view with a "Reset Token" button that changes the view's `.id()` value
- When the reset token changes, the counter should reset to 0 (state destruction via `.id()`)
- Display the current reset token value
- **Manual test:** Increment counter to 5, tap "Reset Token" → counter resets to 0
