### ForEach Refactoring Strategy

**Current `taggedRenderable`** wraps items in `TagModifier(.tag)`:
```swift
private func taggedRenderable(for renderable: Renderable, defaultTag: Any?) -> Renderable {
    if let defaultTag, TagModifier.on(content: renderable, role: .tag) == nil {
        return ModifiedContent(content: renderable, modifier: TagModifier(value: defaultTag, role: .tag))
    } else {
        return renderable
    }
}
```

**New approach ŌĆö dual wrapping (identity + selection tag):**

ForEach must produce BOTH `IdentityKeyModifier` (for structural identity) AND `TagModifier(.tag)` (for selection). The reason: `Picker` and `TabView` read `TagModifier.on(content:role:.tag)` to match selection values against items (see `Picker.swift:147,164,231,340,372` and `TabView.swift:482`). If ForEach stops producing `.tag`, selection matching breaks silently.

The simplest correct approach: ForEach always adds both modifiers. Since `.tag`'s `key()` wrapping in Render is removed (Phase 1 step 7), the `.tag` modifier becomes pure data annotation with negligible overhead.

```swift
/// Wrap a renderable with structural identity and selection tag.
private func identifiedRenderable(for renderable: Renderable, key: Any?) -> Renderable {
    guard let key else { return renderable }
    var result = renderable
    // Add structural identity if not already present
    if result.identityKey == nil {
        result = ModifiedContent(content: result, modifier: IdentityKeyModifier(key: key))
    }
    // Add selection tag if not already present (Picker/TabView need this)
    if TagModifier.on(content: result, role: .tag) == nil {
        result = ModifiedContent(content: result, modifier: TagModifier(value: key, role: .tag))
    }
    return result
}
```

**Critical question ŌĆö `taggedIteration` grouping:** The current `taggedIteration()` wraps multi-renderable iterations (e.g. `CounterCard` + `Divider`) under a single `ComposeView` to avoid duplicate sibling keys. The new `identifiedRenderable` equivalent (`identifiedIteration`) must preserve this grouping behavior ŌĆö group multi-renderable iterations under a single `ComposeView` bearing one `IdentityKeyModifier` + one `TagModifier(.tag)`.

**Evaluation-phase keying restoration:** The current code has evaluation-phase `key()` **removed** (TEMP comments). The recommendation document does NOT specify evaluation-phase keying ŌĆö identity flows through `IdentityKeyModifier` at the producer level and is consumed at the container render loop. The evaluation-phase `key()` in ForEach.Evaluate is therefore **not restored**. The `evaluateKeyed` helper function is dead code ŌĆö `untaggedRenderable` (ForEach.swift:183+) evaluates content directly without calling `evaluateKeyed`. Mark for removal during cleanup.

### Container Loop Migration

**Pattern to apply uniformly:**
```swift
var seenKeys = mutableSetOf<Any>()
for i in 0..<renderables.size {
    let renderable = renderables[i]
    var composeKey: Any = renderable.identityKey ?? i
    if !seenKeys.add(composeKey) {
        composeKey = "\(composeKey)_dup\(i)"
    }
    androidx.compose.runtime.key(composeKey) {
        renderable.Render(context: contentContext)
    }
}
```

**Apply to:**
- VStack v0 loop (`VStack.swift:124-140`) ŌĆö currently uses `composeKey`, migrate to `identityKey`
- VStack v1+ loop (`VStack.swift:158+`) ŌĆö same
- HStack v0 loop ŌĆö same pattern
- HStack v1+ loop ŌĆö same pattern
- ZStack non-animated Box loop (`ZStack.swift:57-60`) ŌĆö currently NO `key()`, add full pattern
- ZStack animated Box loop (`ZStack.swift:92-103`) ŌĆö Phase 2 (gated on AnimatedContent test)
- VStack/HStack AnimatedContent loops ŌĆö Phase 2

**NOT applied to (static children):**
- Toolbar item groups
- NavigationLink labels
- Lazy section headers/footers

### AnimatedContent Concerns

ZStack's `RenderAnimatedContent` at line 71-107 uses:
```swift
contentKey: { $0.map(arguments.idMap) }
```
where `idMap` reads raw `.id()` values via `TagModifier.on(content:role:.id)?.value`. These are `SwiftHashable` ŌĆö the JNI equality problem applies to `contentKey` comparison too.

**Resolution:** Phase 2 must normalize `idMap` output through `normalizeKey()` or switch `contentKey` to use `identityKey`. Document chosen approach.

**The `animateEnterExit` concern:** Paper 07 ┬¦1 identified that `key()` inside `AnimatedContent` content lambda might break the `AnimatedVisibilityScope` chain required by `animateEnterExit`. This must be empirically tested before applying `key()` to animated paths.

### `forEachModifier` Propagation Audit

`IdentityKeyModifier` is found via `forEachModifier` traversal ŌĆö the same mechanism used for `TagModifier`. Current `forEachModifier` implementations:
- `Renderable` default: returns `nil`
- `ModifiedContent.forEachModifier`: checks modifier, then recurses into content (**correct**)
- Must verify: all wrapper types that implement `Renderable` and contain inner content either (a) are leaves or (b) forward `forEachModifier`

This audit is a Phase 1 prerequisite ŌĆö if any wrapper silently drops `forEachModifier`, `identityKey` lookups will return `nil` for affected views.

### Lazy Container Boundary

Lazy containers (`LazyVStack`, `LazyHStack`, `LazyVGrid`, `LazyHGrid`, `List`, `Table`) use `composeBundleString` for key serialization through producer-driven `(Int) -> String` closures. These are **migrated in Plan 04 (Wave 3)** via a `composeBundleNormalizedKey()` adapter that:
1. Wraps `normalizeKey()` to produce String output (resolves type mismatch)
2. Requires equivalence testing before deployment (scroll position stability)
3. Preserves `composeBundleString()` for Navigation.swift route serialisation (NOT identity)

`produceLazyItems()` in ForEach continues to use `taggedRenderable` (not `identifiedRenderable`) for lazy paths. This is acceptable because lazy containers consume keys through their own `items(key:)` API, not through `identityKey`.
