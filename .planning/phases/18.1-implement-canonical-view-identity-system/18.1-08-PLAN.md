**Root cause:** The `idMap` closure in all three container files reads ONLY `TagModifier(.id)`:
```swift
let idMap: (Renderable) -> Any? = { TagModifier.on(content: $0, role: .id)?.value }
```

ForEach's `identifiedRenderable` produces `IdentityKeyModifier` (structural identity) and `TagModifier(.tag)` (selection), but NOT `TagModifier(.id)`. So `idMap` returns `nil` for all ForEach items, `ids` is empty, `ids.size < renderables.size` is always true, and the AnimatedContent path never executes.

**Fix:** Update `idMap` in all three files to fall back to `identityKey` when `TagModifier(.id)` is absent:

```swift
let idMap: (Renderable) -> Any? = { TagModifier.on(content: $0, role: .id)?.value ?? $0.identityKey }
```

This means:
1. If a renderable has an explicit `.id()` modifier (TagModifier with role `.id`), use that value (preserves state-reset semantics)
2. Otherwise, if the renderable has an `IdentityKeyModifier` (from ForEach's `identifiedRenderable`), use its `identityKey` (enables AnimatedContent for ForEach content)
3. If neither is present, `idMap` returns `nil` (unidentified items continue to use the non-animated path)

**Behavioral impact:**
- When ALL renderables in a container have identity (either via `.id()` or `identityKey`), `ids.size == renderables.size`, and the AnimatedContent path executes
- AnimatedContent already handles the non-animated case gracefully: `Animation.current(isAnimating: self.transition.isRunning)` returns `nil` when no animation is active, and the `if let animation` guard on `animateEnterExit` skips transition application
- So items render identically to the non-animated path when no `withAnimation` is active — no visual regression

**Also update the `ids` filtering logic if needed.** After the idMap change, `ids` will contain identityKey values (which are already normalized via `normalizeKey()` at IdentityKeyModifier.init time). The `rememberedIds`, `newIds`, and `rememberedNewIds` sets used for enter/exit animation detection should work correctly with these normalized values — they use standard Set contains/add operations.

**Verify the contentKey normalization is compatible.** The contentKey lambda (from Plan 06) already normalizes through `normalizeKey()`:
```swift
contentKey: { $0.map { r in arguments.idMap(r).map { normalizeKey($0) } } }
```
Since `identityKey` values are ALREADY normalized (at IdentityKeyModifier.init), the `normalizeKey()` call is redundant but harmless (normalizeKey is idempotent for already-normalized values). No change needed to contentKey.
