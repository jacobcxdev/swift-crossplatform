### 6. Lazy Container Identity
expected: Section 6 "Lazy Container Identity (List)". Add and delete cards using the buttons. After changes, scroll position should remain stable (not jump to top). Items should not flicker or duplicate during recomposition. Note: items that scroll off-screen will lose `remember{}` state — this is expected Compose `LazyColumn`/`LazyRow` behaviour (disposal of composition nodes outside viewport). SwiftUI also does not guarantee `@State` preservation for off-screen lazy items. The idiomatic fix for persistent state in lazy containers is state lifting to the parent reducer (standard TCA pattern). The identity system provides stable diff keys via `composeBundleNormalizedKey` but cannot persist state across disposal.
result: PASS (with known limitation) — Add/delete operations work correctly (identity keys are stable for diffing). Scroll position is preserved across mutations. Items that scroll off-screen lose `remember{}` state — this is expected Compose lazy container behaviour, not an identity system bug.

### 7. Transpiler Peer Remembering
expected: Section 7 "Transpiler Peer Remembering". The PeerRememberTestView has a counter and an instance ID. Tap the counter to increment it. Trigger a recomposition (e.g. rotate device, or interact with another section). The counter value and instance ID should survive recomposition — they should NOT reset. Note: tab switching destroys the entire tab's composition scope — all `remember`/`remember(key)` calls are forgotten. This is expected Compose behaviour — `remember` is scoped to the composition node. SwiftUI's `TabView` keeps all tabs in the hierarchy simultaneously; Compose's typical implementation conditionally renders only the selected tab. The identity system is working correctly within its composition scope — the gap is about Compose composition lifecycle, not identity. Future improvement: Skip's `TabView` could render all tabs simultaneously (matching SwiftUI semantics) — filed as separate concern outside Phase 18.1 scope.
result: PASS (with known limitation) — PeerRememberTestView correctly preserves identity within the same composition scope (within a tab). CounterCard correctly preserves identity when adding cards (within the same tab). Tab switching destroys the entire tab's composition scope — this is expected Compose behaviour, not an identity system bug.
