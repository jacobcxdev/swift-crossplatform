The key concern (from Paper 07 section 1) is that key() inside AnimatedContent's content lambda might break the AnimatedVisibilityScope chain required by animateEnterExit.

**Step 1: Normalize contentKey through normalizeKey()**

In all three files, the AnimatedContent receives a `contentKey` parameter derived from `idMap`:
```swift
contentKey: { $0.map(arguments.idMap) }
```
where `idMap` reads raw `.id()` values via `TagModifier.on(content:role:.id)?.value`. These are SwiftHashable — subject to JNI equality issues.

Normalize the idMap output through normalizeKey():
```swift
contentKey: { $0.map { normalizeKey(arguments.idMap($0)) } }
```
Or equivalently, modify idMap to normalize:
```swift
contentKey: { $0.map { renderable in
    let raw = arguments.idMap(renderable)
    return normalizeKey(raw)
} }
```

Apply to:
- VStack.swift AnimatedContent contentKey (~line 201-202)
- HStack.swift AnimatedContent contentKey (~line 166-167)
- ZStack.swift AnimatedContent contentKey (~line 85-86)

**Step 2: Add key(identityKey ?? i) to animated render loops**

Inside the AnimatedContent content lambda, there are render loops that iterate renderables. Apply the same key(identityKey ?? i) + seenKeys guard pattern from Plan 01, but inside the AnimatedContent content lambda.

Apply to:
- VStack.swift animated loop sites (~lines 224-235, 250-261 — check exact locations)
- HStack.swift animated loop sites (~lines 188-199, 214-225 — check exact locations)
- ZStack.swift animated Box loop (~lines 92-103)

The pattern is the same as non-animated loops:
```swift
var seenKeys = mutableSetOf<Any>()
for i in 0..<renderables.size {
    let renderable = renderables[i]
    var composeKey: Any = renderable.identityKey ?? i
    if !seenKeys.add(composeKey) {
        composeKey = "\(composeKey)_dup\(i)"
    }
    androidx.compose.runtime.key(composeKey) {
        renderable.Render(context: contentContext)
    }
}
```
