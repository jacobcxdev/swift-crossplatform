# Phase 18.1: Implement Canonical View Identity System — Gap Closure Research

**Researched:** 2026-03-02
**Domain:** Compose animation lifecycle, lazy container state persistence, transpiler peer remembering hash stability
**Confidence:** HIGH (all findings based on direct codebase analysis)

## Summary

Three gaps were identified during UAT of the Phase 18.1 canonical view identity system. All three have clear root causes in the current codebase with well-defined fix paths:

**GAP-1 (Animated Deletion):** The AnimatedContent path is never entered because `ids.size < renderables.size` is true — ForEach items carry `.id()` tags only when explicitly applied via `.id()` modifier, NOT from ForEach's `identifiedRenderable` which uses `IdentityKeyModifier` (not `TagModifier(.id)`). Since `idMap` reads `TagModifier(.id)` and finds nothing, `ids` is empty, so `ids.size < renderables.size` is always true, and the non-animated Column/Row/Box path executes. The animated path with `animateEnterExit` is never reached.

**GAP-2 (Lazy Container Identity):** Compose's `LazyColumn`/`LazyRow` use `remember{}` for item state, which is disposed when items scroll off-screen. The `composeBundleNormalizedKey` provides stable String keys for Compose's `items(key:)` API, but this only controls item identity for diffing — it does NOT persist state across disposal. CounterCard's internal `Store` and `instanceID` are `let`-with-default properties that get fresh instances on each peer creation. When an item scrolls off and back, the peer is recreated with new Store/UUID.

**GAP-3 (Transpiler Peer Remembering on Tab Switch):** CounterCard has `title: String` as its only constructor param. `Swift_inputsHash` hashes `title` (a value type). When tabs switch, the entire tab content is removed from composition, causing `remember(currentHash)` to be forgotten. On tab return, a fresh peer is created. This is NOT a hash instability issue — it is the expected Compose behaviour when composition nodes are removed. The `remember` call lives inside the tab's composable scope, which is destroyed on tab switch.

**Primary recommendation:** GAP-1 requires rethinking how animated content branching works — the `idMap` needs to read `identityKey` instead of (or in addition to) `TagModifier(.id)`. GAP-2 requires `rememberSaveable` or an external state holder for lazy containers. GAP-3 is fundamentally a Compose composition scope lifecycle issue that requires persisting peer state above the tab switch boundary.

## GAP-1: Animated Deletion — No Visible Exit Animation

### Root Cause Analysis (HIGH confidence)

**The AnimatedContent rendering path is never entered.**

The branching logic in VStack.swift (line 92), HStack.swift (line 89), and ZStack.swift (line 53) is:

```swift
let idMap: (Renderable) -> Any? = { TagModifier.on(content: $0, role: .id)?.value }
let ids = renderables.mapNotNull(idMap)
// ...
if ids.size < renderables.size {
    // NON-ANIMATED path — plain Column/Row/Box loop
} else {
    // ANIMATED path — AnimatedContent with animateEnterExit
}
```

The `idMap` reads `TagModifier` with **role `.id`** — this is the `.id()` modifier used for state reset, NOT the identity key from ForEach. ForEach's `identifiedRenderable` produces `IdentityKeyModifier` (for structural identity) and `TagModifier(.tag)` (for selection), but **never** `TagModifier(.id)`.

For the AnimatedContent path to execute, `ids.size >= renderables.size` — meaning **every** renderable must have a `.id()` tag. In Section 3, the ForEach items have `IdentityKeyModifier` (from `identifiedRenderable`) but no `TagModifier(.id)`. So `ids` is empty, `0 < N` is true, and the non-animated path always executes.

**Why the animated path exists at all:** It was designed for cases where the **developer** explicitly applies `.id()` to every child of a VStack/HStack/ZStack. In that scenario, all renderables have `.id` tags, `ids.size == renderables.size`, and AnimatedContent kicks in. ForEach-based identity (which uses `IdentityKeyModifier`) never triggers this path.

### Fix Strategy

The AnimatedContent branching needs to consider `identityKey` in addition to (or instead of) `TagModifier(.id)`:

**Option A — Use identityKey for animation branching:**
Change `idMap` to read `identityKey` when `.id()` is absent:
```swift
let idMap: (Renderable) -> Any? = {
    TagModifier.on(content: $0, role: .id)?.value ?? $0.identityKey
}
```
This means any ForEach inside a VStack/HStack/ZStack would trigger AnimatedContent when all items have identity keys. However, this changes behaviour for ALL ForEach content — AnimatedContent would become the default rendering path whenever all items have identity keys, even for non-animated mutations.

**Option B — Separate animation trigger from identity source:**
Keep `idMap` as-is for the AnimatedContent branching decision, but provide a way for `withAnimation` blocks to propagate `.id()` tags. In `IdentitySection3View`, the `withAnimation` wrapping the `send(.deleteCardButtonTapped(card.id))` sets `_withAnimation` globally, but the idMap still finds no `.id()` tags.

**Option C (RECOMMENDED) — Use identityKey as animation identity source:**
The core issue is that `idMap` is used for TWO purposes: (1) deciding whether to enter AnimatedContent, and (2) providing content keys for animation diffing. These can be separated:

1. Change the animated-path entry condition to check whether items have `identityKey` OR `.id()`:
   ```swift
   let identityMap: (Renderable) -> Any? = { TagModifier.on(content: $0, role: .id)?.value ?? $0.identityKey }
   let identifiedCount = renderables.count { identityMap($0) != nil }
   if identifiedCount < renderables.size {
       // non-animated path
   } else {
       // animated path — use identityMap for contentKey and entry/exit detection
   }
   ```

2. Inside the animated path, use the combined `identityMap` as the identity source for `contentKey` and `animateEnterExit` entry/exit detection.

**Risk assessment:** Option A is simplest but broadest impact — it makes AnimatedContent the default path for any identified ForEach content. Options B and C are more surgical. The key question is: **should AnimatedContent be used whenever items have identity keys, or only when `.id()` is explicit?**

In SwiftUI, `withAnimation { array.remove(at: i) }` animates removals from `ForEach` automatically. The SwiftUI equivalent would be Option A — ForEach identity enables animation. The AnimatedContent path in Skip already handles the non-animated case gracefully: `Animation.current(isAnimating: self.transition.isRunning)` returns `nil` when there's no active animation, and the `if let animation` guard on `animateEnterExit` skips transition application. So items render identically to the non-animated path when no animation is active.

### Compose Behaviour Explanation

Compose's `AnimatedContent` uses `contentKey` to determine what changed between recompositions. When the key list changes (item removed), it renders both the old and new content simultaneously, applying `animateEnterExit` enter/exit transitions. The container's `SizeTransform` controls how the overall size animates. The `transitionSpec` in the current code uses `EnterTransition.None.togetherWith(ExitTransition.None)` for the container-level transition, relying on per-item `animateEnterExit` modifiers for individual item animations. This design is correct — the issue is purely that this code path is never reached.

### Scope Assessment

This fix is within Phase 18.1 scope — VIEWID-05 and VIEWID-09 specifically cover AnimatedContent normalization and loop keying. The fix completes the intent of the Phase 2 (AnimatedContent) work that was designed and implemented but never properly activated for ForEach-based content.

## GAP-2: Lazy Container Identity — State Resets on Scroll

### Root Cause Analysis (HIGH confidence)

**Compose `remember{}` state is disposed when lazy items scroll off-screen.**

Compose's `LazyColumn` and `LazyRow` aggressively manage composition for performance. When an item scrolls out of the visible viewport (plus a small prefetch buffer), Compose **disposes** its composition entirely — all `remember{}` calls lose their state. When the item scrolls back in, it is recomposed from scratch.

The `composeBundleNormalizedKey` provides stable String keys to Compose's `items(count:key:)` API. This ensures:
- Compose can efficiently diff the list (add/remove/reorder detection)
- Scroll position is preserved across mutations
- The correct item factory is called for the correct index

But it does **NOT** ensure state persistence. The key tells Compose "this is the same item" for diffing, but `remember{}` state is still tied to the composition scope, which is destroyed on scroll-out.

### CounterCard's State Architecture

CounterCard has:
```swift
struct CounterCard: View {
    let title: String           // constructor param
    let store = Store(...)      // let-with-default (new instance per creation)
    let instanceID = UUID()     // let-with-default (new instance per creation)
}
```

The transpiler generates `remember(currentHash)` for the peer, keyed on `Swift_inputsHash(title)`. When the composition scope is destroyed (scroll off), the remembered `SwiftPeerHandle` is forgotten (triggering `onForgotten` -> `Swift_release`). When the item scrolls back, a new peer is created with fresh `Store` and `UUID` instances.

### Fix Strategy

**Option A — `rememberSaveable` for peer handles:**
Replace `remember(key) { SwiftPeerHandle(...) }` with `rememberSaveable(key) { ... }`. However, `rememberSaveable` requires `Saver`/`Parcelable` support, and `SwiftPeerHandle` wraps a raw `Long` peer pointer — saving/restoring a pointer across disposal makes no sense because the Swift object at that address may have been freed.

**Verdict: NOT viable.** Peer pointers cannot be saved/restored.

**Option B — External state holder above lazy scope:**
Store CounterCard state in a map outside the lazy composition scope. This could be:
1. A `remember { mutableMapOf<String, Long>() }` at the `List`/`LazyColumn` parent level
2. An `EnvironmentValues`-based state registry
3. A companion object / singleton registry

This approach would require the transpiler to check whether the view is rendered inside a lazy container and use an external registry instead of (or in addition to) `remember`.

**Verdict: Complex.** Requires significant transpiler changes and runtime detection of lazy context.

**Option C (RECOMMENDED) — Accept as known limitation, document:**
SwiftUI's `List`/`LazyVStack` also doesn't guarantee `@State` preservation for off-screen items — it's an implementation detail whether SwiftUI keeps the state. In practice, SwiftUI often does keep state for recently-scrolled items, but the documentation makes no guarantee. The difference is that Compose is more aggressive about disposal.

For the specific test case (CounterCard with internal `Store`), the idiomatic fix is to **lift state out of the view** — store counter values in the parent `IdentityFeature.State` keyed by card ID, and pass them down. This is the TCA-recommended pattern anyway (single source of truth in the parent reducer).

**Option D — Hybrid: `rememberSaveable` for serializable state only:**
For views with `@State` properties (not let-with-default), the transpiler could use `rememberSaveable` with a `Saver` for the state values themselves (Int, String, etc.), even if the peer pointer can't be saved. The `@State` sync mechanism already knows how to read/write state values. On scroll-back, a new peer would be created but the `@State` values would be restored from the saved bundle.

**Verdict: Viable for `@State` but not for let-with-default Store/UUID.** This addresses `@State var count: Int` but not `let store = Store(...)`.

### Files Potentially Affected

| File | Change |
|------|--------|
| Transpiler (`KotlinBridgeToKotlinVisitor.swift`) | Option D: generate `rememberSaveable` for @State values in lazy context |
| `LazyVStack.swift`, `LazyHStack.swift`, etc. | Option B: external state registry at lazy scope |
| `IdentityFeature.swift` | Option C: lift CounterCard state to parent |

### Scope Assessment

**Option C (document as limitation) is in-scope** for Phase 18.1 — it acknowledges a platform difference rather than requiring deep transpiler changes. **Option D (rememberSaveable for @State)** would be a significant enhancement better suited for a future phase. The test expectation in UAT may need adjustment: "counters should be retained when scrolling" is not guaranteed even in SwiftUI for lazy containers.

## GAP-3: Transpiler Peer Remembering — Tab Switch Loses Identity

### Root Cause Analysis (HIGH confidence)

**Compose `remember(key)` is scoped to the composition node — tab switch destroys the composition.**

CounterCard in Section 7 is rendered inside `IdentitySection7View`, which is inside a `ScrollView` in `IdentityView`. When the user switches tabs (in the app-level TabView, NOT Section 5's mini-TabView), the entire Identity tab content is removed from composition. All `remember` calls within that tab — including `remember(currentHash) { SwiftPeerHandle(...) }` in CounterCard — are forgotten.

The transpiler generates for CounterCard (mixed view: `title` constructor param + `store`/`instanceID` let-with-default):
```kotlin
val currentHash = Swift_inputsHash(Swift_peer)
val peerHandle = remember(currentHash) { SwiftPeerHandle(Swift_peer, ::Swift_retain, ::Swift_release) }
val swapped = peerHandle.peer != Swift_peer
if (swapped) { peerHandle.swapFrom(Swift_peer); Swift_peer = peerHandle.peer }
```

**What `Swift_inputsHash` computes:**
```swift
var hasher = Hasher()
// title is the only constructor param; it's a value type (String), so it IS hashed
if !(type(of: peer_swift.value.title) is AnyClass), let h = peer_swift.value.title as? AnyHashable {
    hasher.combine(h)
}
return Int64(hasher.finalize())
```

The hash is **stable** for the same `title` value. The issue is NOT hash instability. The issue is that the **entire composition scope is destroyed** when tabs switch.

**Why PeerRememberTestView works:** It also uses `remember { SwiftPeerHandle(...) }` (no key variant, Phase 1 unconditional peer remembering). Both PeerRememberTestView and CounterCard would lose state on tab switch for the same reason. The UAT description says "PeerRememberTestView preserves identity on tap" and "CounterCard retains identity when adding cards but loses identity when switching tabs." This means PeerRememberTestView was tested within the same tab (where `remember` survives), while CounterCard was additionally tested across tab switches — exposing a limitation of `remember`-based peer storage.

**Key insight:** The UAT test description for Section 7 says "Trigger a recomposition (e.g. rotate device, or interact with another section)." Interacting with another section (within the same tab) triggers a recomposition but does NOT destroy the composition scope. The user went further and tested **tab switching**, which DOES destroy the composition scope.

### Fix Strategy

**Option A — `rememberSaveable` for peer handles:**
Same analysis as GAP-2 Option A — not viable because peer pointers can't be saved/restored.

**Option B — Lift remember to a scope above the tab:**
Use a `CompositionLocalProvider` or `ViewModel`-scoped map to store peer handles keyed by a stable view identity. This would require:
1. A registry at the `Activity`/`Navigation` scope level
2. A way for the transpiler to generate lookup code against this registry
3. Lifecycle management to clean up entries when views are truly removed (not just tab-switched)

**Verdict: Complex.** Requires runtime infrastructure not currently present.

**Option C (RECOMMENDED) — Accept as known limitation, document:**
This is **expected Compose behaviour**. When a composable is removed from the tree (tab switch, navigation pop, conditional rendering), all its `remember` state is lost. SwiftUI behaves similarly for `@State` in views that are removed from the hierarchy — though SwiftUI's `TabView` internally preserves off-screen tabs' state (they are rendered but hidden), Compose's typical `TabRow` + conditional content approach does not.

The fix at the app level is to either:
1. Use a `TabView` implementation that renders all tabs simultaneously (keeping off-screen tabs in composition)
2. Lift state out of the view into a parent store

**Option D — Skip-UI TabView renders all tabs (keeping them in composition):**
If Skip's `TabView` implementation conditionally renders only the selected tab, changing it to render all tabs (with visibility modifiers) would keep `remember` state alive across tab switches. This is how SwiftUI's native `TabView` works — all tab views exist in the hierarchy simultaneously.

**Verdict: Viable and correct.** This would match SwiftUI's TabView semantics. However, it may have performance implications for tabs with complex content and would be a change to Skip's TabView implementation, not the identity system.

### Files Potentially Affected

| File | Change |
|------|--------|
| `forks/skip-ui/.../Containers/TabView.swift` | Option D: render all tabs, hide non-selected |
| `IdentityFeature.swift` | Option C: document limitation or lift state |
| None (documentation only) | Option C: document as known platform difference |

### Scope Assessment

**Option C (document as limitation)** is in-scope. **Option D (TabView renders all tabs)** is a TabView behaviour change that arguably belongs in a separate phase focused on TabView parity with SwiftUI. The identity system itself is working correctly — `remember(key)` correctly preserves state within the composition scope. The gap is about Compose composition lifecycle, not identity.

## Common Pitfalls

### Pitfall 1: Conflating Identity Keys with Animation Triggers
**What goes wrong:** Assuming `IdentityKeyModifier` (structural identity) and `TagModifier(.id)` (animation/reset trigger) serve the same purpose in the AnimatedContent branching logic.
**Why it happens:** Both are "identity" concepts but serve different roles — `identityKey` is for positional stability in container loops, `.id()` is for state reset and animation content diffing.
**How to avoid:** When modifying the animated path entry condition, be explicit about which identity concept triggers animation.
**Warning signs:** AnimatedContent path never executing despite items having identity keys.

### Pitfall 2: Expecting `remember` to Survive Composition Removal
**What goes wrong:** Assuming `remember{}` or `remember(key){}` persists state when the composable is removed from the tree (scroll off in lazy, tab switch, conditional rendering).
**Why it happens:** In SwiftUI, `@State` often survives view identity changes due to implementation details. Compose is more explicit: `remember` = composition-scoped, `rememberSaveable` = survives process death (via Bundle).
**How to avoid:** For state that must survive composition removal, use `rememberSaveable` (for serializable state) or lift state to a longer-lived scope (ViewModel, parent composable).
**Warning signs:** State resets when scrolling in lazy containers or switching tabs.

### Pitfall 3: Testing Peer Remembering Only Within Stable Composition
**What goes wrong:** Peer remembering tests pass within a tab but fail across tab switches or navigation.
**Why it happens:** `remember`-based peer handles work perfectly within a stable composition scope. The gap only appears when the composition scope is destroyed and recreated.
**How to avoid:** Test peer remembering across composition scope boundaries: tab switches, navigation push/pop, conditional visibility.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Lazy state persistence | Custom remember-like cache | `rememberSaveable` + `Saver` for serializable values, or lift state to parent | Compose has built-in mechanisms for state that must survive disposal |
| Cross-tab state sharing | Manual peer pointer cache | ViewModel-scoped state holders or render all tabs | Compose composition lifecycle is well-defined; work with it, not against it |
| Animation branching | Complex conditional animation logic | Let AnimatedContent handle enter/exit via its built-in diffing | AnimatedContent + `animateEnterExit` is the canonical Compose animation pattern |

## Architecture Patterns

### Pattern 1: AnimatedContent Entry Condition
**What:** The condition for entering the AnimatedContent rendering path should be based on whether items have usable identity (from any source), not just `.id()` tags.
**When to use:** Whenever a container (VStack/HStack/ZStack) renders ForEach content that may be animated.

### Pattern 2: State Lifting for Lazy Containers
**What:** State that must survive lazy container disposal should be stored in the parent/reducer, not in the child view.
**When to use:** Any time a view inside a `List`/`LazyVStack`/`LazyHStack` needs persistent state.

### Anti-Patterns to Avoid
- **Saving peer pointers to `rememberSaveable`:** Peer pointers are raw memory addresses — they cannot be serialized/deserialized.
- **Assuming composition scope = view lifetime:** In Compose, composition scope is tied to the composable's presence in the active tree, which is narrower than SwiftUI's view lifetime for lazy containers and tab views.

## Open Questions

1. **Should ForEach identity (via `identityKey`) automatically enable AnimatedContent?**
   - What we know: SwiftUI animates ForEach changes automatically when wrapped in `withAnimation`. The AnimatedContent path implements this but is gated on `.id()` tags.
   - What's unclear: Whether making AnimatedContent the default for identified content would cause unintended visual changes in existing apps.
   - Recommendation: Implement Option A (use identityKey in idMap) and test empirically. If visual regressions occur, gate on `Animation.isInWithAnimation`.

2. **Should Skip's TabView render all tabs simultaneously?**
   - What we know: SwiftUI TabView keeps all tab views in the hierarchy. Compose TabRow + conditional content does not.
   - What's unclear: Performance impact of rendering all tabs; whether Skip's TabView already has this capability.
   - Recommendation: Investigate Skip's TabView implementation to determine if this is a simple flag change or a significant refactor. File as separate issue if complex.

3. **Is lazy container state preservation within Phase 18.1 scope?**
   - What we know: The current test expects counters to survive scrolling, which requires either state lifting or `rememberSaveable`.
   - What's unclear: Whether the UAT expectation should be adjusted vs implementing a fix.
   - Recommendation: Document as known platform difference. Adjust UAT expectation for Section 6 to note that lazy state preservation requires state lifting (TCA pattern). Consider `rememberSaveable` for `@State` in a future phase.
