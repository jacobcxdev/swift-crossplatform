# Phase 18.1: Implement Canonical View Identity System - Research

**Researched:** 2026-03-01
**Domain:** Compose view identity architecture in SkipUI (skip-ui fork + skipstone transpiler)
**Confidence:** HIGH

## Summary

Phase 18.1 implements the canonical view identity system as specified in the reconciled recommendation document (`docs/identity-proposals/00-final-recommendation.md`), refined by CONTEXT.md decisions. The work is entirely within the `forks/skip-ui` and `forks/skipstone` codebases, touching approximately 11 files with ~200 lines net change. All code is `#if SKIP`-gated — zero iOS impact.

The architecture decouples structural identity (`IdentityKeyModifier`) from selection tagging (`TagModifier(.tag)`), introduces a single normalisation function (`normalizeKey()`) replacing the fragile `composeKeyValue()`, and applies uniform container loop keying with duplicate-key guarding. The phasing (0 through 3) is sequential with Phase 3 (transpiler) being parallelisable.

**Primary recommendation:** Follow the CONTEXT.md phasing exactly — Phase 0 immediate fixes, Phase 1 core architecture as one atomic change, Phase 2 AnimatedContent empirically gated on tests, Phase 3 transpiler fix independently.

**Rationale for keeping new types in `Renderable.swift`:** `normalizeKey()`, `AnyOptionalProtocol`, `IdentityKeyModifier`, and the `identityKey`/`selectionTag` extensions are all logically part of the Renderable identity system. Keeping them together in `Renderable.swift` mirrors how `composeKey` and `composeKeyValue()` already live there. The file is currently 93 lines — adding ~60 lines keeps it well under 200 lines total.

### Pattern 1: Producer-Side Normalisation
**What:** All key normalisation happens once at the producer (`IdentityKeyModifier.init`). Consumers never normalise.
**When to use:** Always — this is a design invariant.
**Example:**
```swift
// Renderable.swift — normalizeKey() called once at init
final class IdentityKeyModifier: RenderModifier {
    let normalizedKey: Any  // String | Int | Long guaranteed
    init(key: Any) {
        self.normalizedKey = normalizeKey(key)
        super.init(role: .unspecified)
    }
    @Composable override func Render(content: Renderable, context: ComposeContext) {
        content.Render(context: context)  // transparent
    }
}
```

### Pattern 2: Dual Wrapping in ForEach
**What:** ForEach produces BOTH `IdentityKeyModifier` (structural identity) AND `TagModifier(.tag)` (selection).
**When to use:** All non-lazy ForEach paths.
**Why dual:** Picker reads `TagModifier.on(content:role:.tag)` at lines 147, 164, 231, 340, 372. TabView reads it at line 482. Removing `.tag` without providing it via another mechanism breaks selection silently.
**Example:**
```swift
// ForEach.swift — replaces taggedRenderable
private func identifiedRenderable(for renderable: Renderable, key: Any?) -> Renderable {
    guard let key else { return renderable }
    var result = renderable
    if result.identityKey == nil {
        result = ModifiedContent(content: result, modifier: IdentityKeyModifier(key: key))
    }
    if TagModifier.on(content: result, role: .tag) == nil {
        result = ModifiedContent(content: result, modifier: TagModifier(value: key, role: .tag))
    }
    return result
}
```

### Pattern 3: Container Loop with Duplicate-Key Guard
**What:** Uniform `key(identityKey ?? i)` pattern with `seenKeys` set preventing duplicate keys.
**When to use:** All eager container render loops with potentially-dynamic children.
**Example:**
```swift
// VStack/HStack/ZStack render loops
var seenKeys = mutableSetOf<Any>()
for i in 0..<renderables.size {
    let renderable = renderables[i]
    var composeKey: Any = renderable.identityKey ?? i
    if !seenKeys.add(composeKey) {
        composeKey = "\(composeKey)_dup\(i)"
    }
    androidx.compose.runtime.key(composeKey) {
        renderable.Render(context: contentContext)
    }
}
```

### Pattern 4: TagModifier Simplification
**What:** `.tag` role becomes pure data annotation (no `key()` in Render). `.id` role normalises through `normalizeKey()`.
**When to use:** Phase 1 step 7, gated on render loop audit.
**Example:**
```swift
// TagModifier.Render simplified
@Composable override func Render(content: Renderable, context: ComposeContext) {
    if let stateSaver {
        var context = context
        context.stateSaver = stateSaver
        let idKey = normalizeKey(value ?? Self.defaultIdValue)
        androidx.compose.runtime.key(idKey) {
            super.Render(content: content, context: context)
        }
    } else {
        // .tag role: NO key() — pure data annotation
        super.Render(content: content, context: context)
    }
}
```

### Anti-Patterns to Avoid
- **Double normalisation:** Never call `normalizeKey()` at both producer and consumer. The producer (`IdentityKeyModifier.init`) normalises once; `identityKey` returns the already-normalised value.
- **Conditional `key()` wrapping:** `key()` MUST be at the top level of loop bodies, never inside `if let`. Wrapping `key()` in a conditional creates per-iteration replaceable groups that prevent Compose from matching movable key groups across iterations.
- **Using `composeKey` in new code:** `composeKey` and `composeKeyValue()` are deprecated. All new code uses `identityKey` and `normalizeKey()`.
- **Restoring evaluation-phase `key()` in ForEach:** The TEMP-removed evaluation-phase keying is intentionally not restored. Identity flows through `IdentityKeyModifier` at the producer level.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Optional unwrapping for keys | String-based `"Optional(...)"` stripping | `AnyOptionalProtocol` with structural unwrapping | String-based approach is fragile, locale-dependent |
| Key type safety | Type-safe enum wrapper | `normalizeKey()` returning `Any` | Compose's `key()` accepts `Any`; enum adds boilerplate for no runtime benefit |
| Modifier traversal | Recursive `ModifiedContent` unwrapping | `forEachModifier` (existing mechanism) | Already works for `TagModifier`; `IdentityKeyModifier` uses the same path |
| Duplicate key deduplication | Custom dedup logic | `mutableSetOf<Any>` + `.add()` return value | Kotlin's `MutableSet.add()` returns `false` if element already present — clean one-liner |

**Key insight:** The existing `forEachModifier` traversal in `ModifiedContent` (line 258) is the critical infrastructure that makes the modifier-based approach work. It checks the modifier first, then recurses into content. This means `IdentityKeyModifier` is automatically discoverable through any depth of `ModifiedContent` wrapping without any new traversal code.

## Common Pitfalls

### Pitfall 1: TagModifier `.tag` Role `key()` Removal Without Audit
**What goes wrong:** Removing `.tag` role's `key()` wrapping in `TagModifier.Render` (line 1429-1439) before confirming no other container depends on it causes silent identity loss in those containers.
**Why it happens:** Picker iterates tagged renderables in segmented, menu, and navigationLink styles. If any of these depend on `.tag`'s `key()` for positioning (not just selection matching), removing it breaks identity.
**How to avoid:** Phase 1 step 7 is explicitly gated on auditing: Picker.swift (4 sites), TabView.swift, Menu.swift. For each, confirm it only reads `.tag` as data (selection matching) or has its own container-level keying.
**Warning signs:** Selection stops working in Picker after Phase 1.

### Pitfall 2: `forEachModifier` Propagation Gaps
**What goes wrong:** If any `Renderable` wrapper type does not forward `forEachModifier`, `identityKey` returns `nil` for views wrapped by it, causing fallback to positional identity.
**Why it happens:** New wrapper types (or existing ones not audited) that implement `Renderable` but don't override `forEachModifier` use the default implementation which returns `nil`.
**How to avoid:** Phase 1 step 10 audits all `Renderable` conformers. Current non-forwarding conformers identified:
- `LazySectionHeader` — no `forEachModifier` override (leaf-like, wraps list of renderables)
- `LazySectionFooter` — same
- `ViewRenderable` — no `forEachModifier` override (wraps a `View`, not a `Renderable`)
These are all used in lazy/static contexts where `identityKey` is not consumed, so no gap. But the audit must verify this.
**Warning signs:** Items in specific container types lose state on reorder/deletion.

### Pitfall 3: AnimatedContent `contentKey` Using Raw SwiftHashable
**What goes wrong:** The `contentKey` parameter in AnimatedContent uses `idMap` which reads raw `.id()` values. These are `SwiftHashable` — subject to the JNI equality problem that the entire identity system is designed to solve.
**Why it happens:** `contentKey: { $0.map(arguments.idMap) }` in VStack (line 201-202), HStack (line 166-167), and ZStack (line 85-86) passes raw tag values to Compose's diff algorithm.
**How to avoid:** Phase 2 normalises `idMap` output through `normalizeKey()` or switches `contentKey` to use `identityKey`.
**Warning signs:** Animated transitions fire incorrectly or items don't animate on deletion.

### Pitfall 4: `identifiedIteration` Multi-Renderable Grouping
**What goes wrong:** When a ForEach iteration produces multiple renderables (e.g. `CounterCard` + `Divider`), failing to group them under a single `ComposeView` with one `IdentityKeyModifier` creates duplicate sibling keys.
**Why it happens:** The current `taggedIteration` handles this by wrapping multiple renderables in a `ComposeView`. The replacement `identifiedIteration` must preserve this exact behaviour.
**How to avoid:** Directly port the grouping logic from `taggedIteration` to `identifiedIteration`, replacing `TagModifier(.tag)` wrapping with `IdentityKeyModifier` + `TagModifier(.tag)` dual wrapping.
**Warning signs:** Compose crashes or warns about duplicate keys in VStack/HStack containers.

### Pitfall 5: Transpiler `stateVariables.isEmpty` Guard
**What goes wrong:** Views with both `@State` properties AND `let`-with-default properties don't get peer remembering because the guard at `KotlinBridgeToKotlinVisitor.swift:1734` skips the `Evaluate`/`_ComposeContent` override generation when `stateVariables` is non-empty.
**Why it happens:** The guard was written assuming `@State` views don't need peer remembering. But mixed views (e.g. `@State var count = 0` + `let instanceID = UUID()`) need both state management AND peer remembering.
**How to avoid:** Phase 3 removes the `stateVariables.isEmpty` condition, allowing peer remembering to coexist with state variables.
**Warning signs:** Views with both `@State` and `let`-with-default recreate their `let` properties on every recomposition.

## `forEachModifier` Propagation Audit (Pre-Research)

Direct `Renderable` conformers and their `forEachModifier` status:

| Type | File | Has Inner Content | Forwards `forEachModifier`? | Risk |

**Assessment:** `LazySectionHeader`, `LazySectionFooter`, and `ViewRenderable` do not forward `forEachModifier`. However:
- `LazySectionHeader`/`LazySectionFooter` are only used in lazy container contexts (List section headers/footers) where `identityKey` is not consumed by container loops.
- `ViewRenderable` wraps a `View` for deferred rendering — it's created internally and not placed inside ForEach-produced modifier chains.

No gap requires remediation for Phase 18.1 scope. The planner should include this audit as a verification step.

## Open Questions

1. **AnimatedContent `key()` + `animateEnterExit` interaction (Phase 2)**
   - What we know: Paper 07 §1 identified theoretical concern that `key()` inside `AnimatedContent`'s content lambda might break the `AnimatedVisibilityScope` chain required by `animateEnterExit`. Both synthesis documents agree empirical testing is required.
   - What's unclear: Whether Compose's `key()` inside `AnimatedContent` content lambda preserves `AnimatedVisibilityScope`.
   - Recommendation: Phase 2 writes the test FIRST. If test fails, investigate `contentKey` enrichment as alternative.

2. **Container loop helper extraction**
   - What we know: The `seenKeys` + `key(identityKey ?? i)` pattern is identical across 6+ loop sites (VStack v0, VStack v1+, HStack v0, HStack v1+, ZStack non-animated, potentially ZStack animated).
   - What's unclear: Whether extracting to a helper function creates non-inline function group boundaries that break Compose's movable-group matching (same issue documented in `evaluateKeyed` comments at ForEach.swift:87-94).
   - Recommendation: Inline the pattern in each loop. The code is 6 lines and repetition is safer than risking Compose's group mechanics. This is Claude's discretion per CONTEXT.md.

3. **`explicitResetKey` promotion (Phase 2 checkpoint)**
   - What we know: `.id()` reset scoping remains implicit via `TagModifier(.id)`'s nested `key()`. For non-animated eager containers, this nested approach has no known failure mode.
   - What's unclear: Whether AnimatedContent diff logic can distinguish sibling matching from state reset with the nested key-group approach.
   - Recommendation: Defer decision to Phase 2. If AnimatedContent tests fail, promote `explicitResetKey` to a field on `IdentityKeyModifier`.
