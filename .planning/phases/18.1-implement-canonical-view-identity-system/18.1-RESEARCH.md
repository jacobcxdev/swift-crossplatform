## GAP-R2-1: ConcurrentModificationException in RetainedAnimatedItems.sync (CRITICAL)

### Root Cause Analysis (HIGH confidence)

**File:** `forks/skip-ui/Sources/SkipUI/SkipUI/Containers/RetainedAnimatedItems.swift`, lines 119-128.

Step 3 of `sync()` iterates `items` (a `mutableMapOf<Any, RetainedAnimatedItem>()`, which transpiles to Kotlin `LinkedHashMap`) using `for (key, item) in items` and calls `items.remove(key)` inside the loop body when an item should be removed immediately (no animation):

```swift
// Step 3: Mark removed items for exit or remove immediately
for (key, item) in items {
    if !currentKeySet.contains(key) && item.visibility.targetState == true {
        if let animation {
            item.visibility.targetState = false
            item.animation = animation
        } else {
            items.remove(key)  // <-- CRASH: mutating during iteration
        }
    }
}
```

Kotlin's `LinkedHashMap` iterator throws `ConcurrentModificationException` when the underlying map is modified during iteration. This is a well-known Kotlin/Java collection constraint.

### Fix (HIGH confidence)

Collect keys to remove into a separate list, then remove them after the iteration completes. This is the **same pattern already used in Step 4** of the same function:

```swift
// Step 3: Mark removed items for exit or collect for immediate removal
var immediateRemovalKeys = mutableListOf<Any>()
for (key, item) in items {
    if !currentKeySet.contains(key) && item.visibility.targetState == true {
        if let animation {
            item.visibility.targetState = false
            item.animation = animation
        } else {
            immediateRemovalKeys.add(key)
        }
    }
}
for key in immediateRemovalKeys {
    items.remove(key)
}
```

**Scope:** Trivial fix — 4 lines changed. No architectural impact.

## GAP-R2-3: Tab Switch Peer Survival Not Working

### Root Cause Analysis (MEDIUM confidence — needs runtime diagnosis)

TabView.swift has correct PeerStore integration:
- Line 386: `let tabPeerStore = remember { PeerStore() }` — created outside NavHost, survives tab switches
- Line 387-388: `CompositionLocalProvider(providedPeerStore)` — wraps NavHost
- Line 417-418: `LocalPeerStoreNamespace provides tabIndex.toString()` — per-route namespace

The `PeerRememberTestView` in Section 7 is rendered directly inside the ScrollView of the Identity tab — it is NOT inside a TabView. Section 7 tests transpiler peer remembering within a single composition scope. The UAT failure "PeerRememberTestView loses identity on tab switch" means switching away from the **app-level** Identity tab (not Section 5's mini-TabView).

**Hypothesis A (LIKELY): App-level TabView vs Section 5 mini-TabView confusion.**
The app uses a TabView for its main navigation (Basics / Identity / etc). When the user switches from Identity tab to another tab, the entire Identity tab content is removed from NavHost composition, destroying ALL `remember`-scoped state. The PeerStore added in Plan 10 is on **Section 5's mini-TabView**, not the app-level TabView. If the app-level TabView also uses NavHost internally, it would need its own PeerStore wrapping.

**Hypothesis B: Missing IdentityKeyModifier on PeerRememberTestView content.**
`rememberViewPeer()` requires `LocalPeerStoreItemKey` (set by `IdentityKeyModifier`) to be non-nil for the PeerStore path to activate (the `if let store, let itemKey` gate at PeerStore.swift line 199). If PeerRememberTestView is not inside a ForEach (which provides IdentityKeyModifier via `identifiedRenderable`), then `itemKey` is nil and the fallback `remember`-based path is used, which doesn't survive composition disposal.

**Hypothesis C: NavHost popUpTo(saveState: true) not preserving CompositionLocal state.**
Even with PeerStore outside NavHost, if the NavHost's `composable` blocks destroy and recreate their composition trees on tab switch, the `LocalPeerStoreItemKey` and `LocalPeerStoreNamespace` CompositionLocals would need to be re-provided. The PeerStore itself survives (it's outside NavHost), but the lookup keys depend on CompositionLocals that are re-established on recomposition.

### Recommended Investigation

1. Add diagnostic logging to `rememberViewPeer()` showing: `store != null`, `itemKey`, `namespace`, `cacheKey`, hit/miss
2. Check if `PeerRememberTestView` is inside a ForEach or has an explicit `IdentityKeyModifier` (likely not — it appears to be a standalone view)
3. Determine whether the UAT failure is about the app-level TabView (not Section 5) — if so, the app-level TabView needs PeerStore integration or the test expectation needs adjustment

### Fix Strategy

**If Hypothesis A (app-level TabView):** The fix is either:
- Add PeerStore to the app-level TabView (same pattern as Section 5's mini-TabView)
- Accept that app-level tab switches destroy composition and adjust test expectations
- Note: This is the same behaviour as SwiftUI when a TabView's content is not kept alive

**If Hypothesis B (missing item key):** The fix is to ensure PeerRememberTestView gets an item key. Options:
- Wrap it in a ForEach (unusual for a single view)
- Provide a manual `LocalPeerStoreItemKey` in Section 7's view hierarchy
- Add a `peerStoreKey` view modifier that sets `LocalPeerStoreItemKey`

**If Hypothesis C (NavHost recomposition):** The architecture may need CompositionLocal restoration after NavHost route switch, which is a deeper issue.

### Scope Assessment

Hypothesis B is the most likely and cheapest to fix — provide a manual item key for standalone views that need PeerStore integration. This could be a simple view modifier or explicit CompositionLocalProvider.

## Common Pitfalls

### Pitfall 1: Kotlin Collection Mutation During Iteration
**What goes wrong:** Iterating a Kotlin `MutableMap` or `MutableList` with `for ... in` and calling `remove()` inside the loop throws `ConcurrentModificationException`.
**Why it happens:** Swift allows mutation during iteration (copies the collection or uses stable iterators). Kotlin/Java collections use fail-fast iterators that detect structural modification.
**How to avoid:** Always collect keys/indices to remove into a separate list, then remove after the loop. Or use `iterator.remove()` via the iterator API.
**Warning signs:** Any `for (key, value) in mutableMap { ... mutableMap.remove(key) }` pattern in Swift code that will be transpiled to Kotlin.

### Pitfall 2: PeerStore Gate Requires Both store AND itemKey
**What goes wrong:** `rememberViewPeer()` falls back to `remember`-based path even when a PeerStore is in scope.
**Why it happens:** The gate at PeerStore.swift line 199 is `if let store, let itemKey`. If `LocalPeerStoreItemKey` is nil (view is not inside a ForEach or has no IdentityKeyModifier), the PeerStore path is skipped entirely.
**How to avoid:** Ensure views that need PeerStore-backed peer survival are either (a) inside a ForEach (which provides IdentityKeyModifier -> LocalPeerStoreItemKey), or (b) have an explicit LocalPeerStoreItemKey provider in their ancestor hierarchy.
**Warning signs:** Standalone views (not in ForEach) losing state despite being inside a PeerStore-providing container.

### Pitfall 3: Confusing App-Level TabView with Section-Level TabView
**What goes wrong:** PeerStore integration on a nested TabView (Section 5) doesn't help with app-level tab switching.
**Why it happens:** The app-level TabView and Section 5's mini-TabView are separate composition scopes. PeerStore on one doesn't affect the other.
**How to avoid:** When testing tab switch peer survival, be explicit about WHICH TabView is being switched. App-level tab switches destroy the entire Identity tab composition.

### Pitfall 4: Shared State Across Test Sections
**What goes wrong:** Adding/deleting items in one section affects unrelated sections, causing confusing scroll jumps and state pollution.
**Why it happens:** Multiple sections read from the same TCA state array.
**How to avoid:** Each test section should have its own independent data array in the TCA state.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Collection filtering with removal | `for (k,v) in map { map.remove(k) }` | Collect-then-remove or `entries.removeAll { predicate }` | Kotlin iterator invalidation |
| Standalone view peer survival | Manual remember + external cache | Provide `LocalPeerStoreItemKey` via view modifier or explicit CompositionLocalProvider | PeerStore already handles caching; just needs the lookup key |
| Per-section test data | Shared mutable array with section filters | Per-section arrays in TCA state | Clean isolation, no cross-contamination |

## Architecture Patterns

### Pattern 1: Collect-Then-Remove for Kotlin Maps
**What:** When iterating a mutable map and conditionally removing entries, collect removal keys into a list, then iterate the list to remove.
**When to use:** Any `for (key, item) in mutableMap` loop that may call `mutableMap.remove()`.
**Example:**
```swift
var toRemove = mutableListOf<Any>()
for (key, item) in items {
    if shouldRemove(key, item) {
        toRemove.add(key)
    }
}
for key in toRemove {
    items.remove(key)
}
```

### Pattern 2: PeerStore Item Key for Standalone Views
**What:** When a view needs PeerStore-backed peer survival but isn't inside a ForEach, provide a manual item key.
**When to use:** Standalone views inside TabView routes or lazy containers that need to survive composition disposal.
**Example:**
```swift
// In the parent view providing PeerStore:
CompositionLocalProvider(LocalPeerStoreItemKey provides AnyHashable("standalone-view-key")) {
    StandaloneView()
}
```

### Anti-Patterns to Avoid
- **Mutating Kotlin collections during iteration:** Always collect-then-remove.
- **Assuming all views inside PeerStore containers automatically use PeerStore:** The `rememberViewPeer()` gate requires non-nil `itemKey`.
- **Testing identity across composition scope boundaries without PeerStore at every boundary:** Each TabView/NavHost boundary needs its own PeerStore wrapping.
