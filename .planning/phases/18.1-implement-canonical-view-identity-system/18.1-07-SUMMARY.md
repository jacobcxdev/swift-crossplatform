## Key Equivalence Analysis

Verified that composeBundleNormalizedKey() produces identical String output as composeBundleString() for all common ForEach identifier types:

| Type | composeBundleString | composeBundleNormalizedKey | Match |
|------|-------------------|---------------------------|-------|
| String "hello" | "hello" | "hello" | Yes |
| Int 42 | "42" | "42" | Yes |
| UUID | UUID string | UUID string | Yes |
| Identifiable (String id) | id string | id string | Yes |
| Identifiable (UUID id) | UUID string | UUID string | Yes |
| Identifiable (Int id) | int string | int string | Yes |
| RawRepresentable (String) | rawValue | rawValue | Yes |
| RawRepresentable (Int) | int string | int string | Yes |
| Optional.some("test") | "Optional(\"test\")" | "test" | **Diverges** |
| nil | "nil" | "nil" | Yes |

**Optional divergence:** composeBundleNormalizedKey unwraps Optionals (producing the inner value string), while composeBundleString wraps them as "Optional(...)". This is acceptable because ForEach identifiers return non-optional values for object-based ForEach (the common case). The normalized form is actually more correct.

## Decisions Made
- composeBundleNormalizedKey wraps normalizeKey() then converts to String -- thin adapter preserves the `(Int) -> String` type signature required by Compose's lazy items() API
- Optional handling divergence accepted as improvement -- normalizeKey unwraps while composeBundleString wraps; ForEach identifiers are non-optional in practice
- composeBundleString retained (not deprecated) -- Navigation.swift uses it for route serialisation
