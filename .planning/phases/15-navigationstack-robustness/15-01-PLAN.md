In `NavigationStack+Observation.swift`, locate the `_TCANavigationStack.body` computed property where the `androidPath` `Binding<NavigationPath>` is created (around line 236-249). The `set:` closure currently has:

```swift
if newPath.count > currentCount {
  // Push: ...
  // Note: NavigationPath doesn't expose elements, so push is handled
  // by navigationDestination(for:) callback, not by path set
}
```

Replace the no-op push branch with actual push dispatch:
1. When `newPath.count > currentCount`, extract the last element from the new path using `newPath[newPath.count - 1]` (skip-fuse-ui's `NavigationPath` has a subscript returning `AnyHashable`).
2. The pushed value arrives as a `StackState<State>.Component` wrapped in `AnyHashable`. Cast it: `if let component = newPath[newPath.count - 1] as? StackState<State>.Component`.
3. Dispatch `store.send(.push(id: component.id, state: component.element))`.
4. If the cast fails, this is a non-TCA push ΓÇË leave it as a no-op (skip-fuse-ui handles it natively).

**Important:** The `SwiftHashable` wrapping may affect the cast chain. On Android, the value may arrive as `SwiftHashable` Γ¶Ê `.base` Γ¶Ê `AnyHashable` Γ¶Ê `StackState.Component`. Test the round-trip empirically. If direct `as? StackState<State>.Component` fails, try `(newPath[newPath.count - 1] as? SwiftHashable)?.base as? StackState<State>.Component`. Note: the `NavigationPath.subscript` returns `AnyHashable`, and the `setData` closure in skip-fuse-ui already unwraps `SwiftHashable` before building the path (`($0 as! SwiftHashable).base as! AnyHashable`), so by the time the `set:` closure receives the new path, elements should already be plain `AnyHashable`-wrapped `Component` values.
