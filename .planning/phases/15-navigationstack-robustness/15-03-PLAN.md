**Phase 1: Diagnose the root cause.**

The dismiss pipeline in `PresentationReducer.swift` (lines 688-694) is:
```swift
presentEffects = .concatenate(
    .publisher { Empty(completeImmediately: false) }
        ._cancellable(id: PresentationDismissID(), navigationIDPath: ...),
    .publisher { Just(self.toPresentationAction.embed(.dismiss)) }
)
```

When `DismissEffect` calls `Task._cancel(id: PresentationDismissID())`, it cancels the `Empty` publisher. The `Concatenate` subscriber then subscribes to `Just(.dismiss)`, which fires immediately.

**Investigation steps (do these in order):**
1. Write a focused test in `PresentationTests.swift` that creates a minimal `@Reducer` with `@Presents` optional child state, sends a child action that triggers `await dismiss()`, and asserts `store.receive(\.destination.dismiss)`. Run on Darwin first to establish baseline.
2. Check if the issue is in the `_cancellable` implementation — specifically whether `Task._cancel` correctly resolves the `navigationIDPath` dependency on Android. The `Dismiss.swift` Android fallback path calls `Task._cancel(id:navigationID:)` — verify the navigation ID path is populated correctly.
3. Check if OpenCombine's `Empty` cancellation propagates correctly through `Concatenate`. If the concatenation operator doesn't properly handle cancellation of the first upstream as a signal to subscribe to the second, that's the root cause.

**Phase 2: Apply the fix.**

Based on diagnosis, apply one of these approaches (in order of preference — use the least invasive fix that works):

**Approach A — Fix OpenCombine concatenation (if that's the root cause):**
If `Concatenate` doesn't properly handle cancellation → completion → subscribe-to-next, fix it in the OpenCombine fork. This is the cleanest fix as it preserves TCA's upstream publisher pattern.

**Approach B — Replace publisher-based dismiss with Effect.run (if publisher cancellation is fundamentally broken on Android):**
Replace the `Empty + Just` concatenation with an `Effect.run`-based approach:
```swift
presentEffects = Effect.run { send in
    await withTaskCancellationHandler {
        // Keep alive until cancelled
        let stream = AsyncStream<Never> { _ in }
        for await _ in stream {}
    } onCancel: { }
    // After cancellation, send dismiss
    await send(self.toPresentationAction.embed(.dismiss))
}
._cancellable(id: PresentationDismissID(), navigationIDPath: ...)
._cancellable(navigationIDPath: ...)
._cancellable(id: OnFirstAppearID(), navigationIDPath: .init())
```
This approach uses Swift concurrency's `withTaskCancellationHandler` which works reliably across JNI.

**Approach C — Add explicit yield after cancellation (if timing is the issue):**
If the cancel propagates but the emit happens before the subscriber is ready, add `Task.yield()` between the cancel and the emit to allow the async boundary to settle.
