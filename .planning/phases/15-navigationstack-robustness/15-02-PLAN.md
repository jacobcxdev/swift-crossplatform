<objective>
Fix JVM type erasure for multi-destination NavigationStack: embed the generic Element type name in the destination key at Swift compile time so that multiple `navigationDestination(for: StackState<X>.Component.self)` registrations produce distinct keys on JVM.

Purpose: On JVM, generic type parameters are erased at runtime, so `StackState<A>.Component` and `StackState<B>.Component` produce identical `String(describing:)` keys. This causes the last registration to overwrite all previous ones, breaking multi-destination stacks.

Output: Type-discriminating destination key in NavigationStack+Observation.swift + tests
</objective>

The fix must ensure that destination keys for `StackState<A>.Component` and `StackState<B>.Component` produce DIFFERENT keys, even when JVM erases the generic parameter at runtime.

**Strategy: Embed Element type name at Swift compile time.**

In `NavigationStack+Observation.swift`, locate `_NavigationDestinationViewModifier` where `.navigationDestination(for: StackState<State>.Component.self)` is called (around line 211). The key needs to include the `State` type name.

**Approach (determine the best option):**

**Option A — Custom `CustomStringConvertible` on Component:**
Add a `destinationKey` computed property to `StackState.Component` that produces `"Component<\(String(describing: Element.self))>"`. This is captured at Swift compile time where `Element` is known. Then, in `_TCANavigationStack`, override the `destinationKeyTransformer` to use this property when the value is a `StackState.Component`.

However, `_TCANavigationStack` may not have direct control over the `destinationKeyTransformer` — that's set inside skip-fuse-ui's `NavigationStack.Java_view`. The TCA adapter needs to propagate the key.

**Option B — Override `description` on Component:**
Make `StackState.Component` conform to `CustomStringConvertible` with `description` returning `"Component<\(String(describing: Element.self))>"`. Since skip-fuse-ui uses `String(describing: type(of: value))` for the pushed VALUE and `String(describing: data)` for the registration TYPE, both sides need to match.

Registration side: `String(describing: StackState<State>.Component.self)` — this calls `String(describing:)` on the TYPE. To make this include the generic param, override the static description.

Lookup side: `String(describing: type(of: componentInstance))` — this gets the TYPE of an INSTANCE.

On JVM both produce the same erased string. The fix must happen at a level where Swift type info is preserved.

**Recommended approach:**
1. Add a stored `static let _destinationKey: String = "StackState.Component<\(String(describing: Element.self))>"` to `StackState.Component` or compute it in `_NavigationDestinationViewModifier`.
2. In `_NavigationDestinationViewModifier.body`, use the explicit `destinationKey` parameter: `.navigationDestination(destinationKey: "StackState.Component<\(String(describing: State.self))>", ...)` if skip-fuse-ui supports it.
3. In `_TCANavigationStack.body`, set the `destinationKeyTransformer` (if exposed by skip-fuse-ui) to extract the element type name from `StackState.Component` instances. If not directly exposed, the pushed Component needs to carry its type name so `String(describing: type(of:))` works correctly — this means the `Component` struct needs a stored `elementTypeName` and its `CustomStringConvertible` should return it.

**Check skip-fuse-ui's `NavigationStack.init(path:...)` for how `destinationKeyTransformer` is injected.** If skip-fuse-ui passes it to skip-ui as a parameter, TCA can override it via a view modifier or environment value. If not, the fix must go through `Component`'s type description.

Also check `NavigationStackStore.swift` (UIKit path) for the same pattern — it may need a matching fix for consistency, though the UIKit path is less likely used on Android.

**Tests to add in `NavigationStackTests.swift`:**

**Test 3: `testDestinationKeyIncludesTypeName`** — Assert that the destination key for `StackState<SomeState>.Component` contains the string "SomeState".

All tests must be platform-agnostic (no `#if os(Android)` gates).
