Using the requirement-evidence-map.md from Plan 01, update the REQUIREMENTS.md traceability table:

1. **Add Evidence column** to the traceability table. Change the table header from:
   ```
   | Requirement | Phase | Status |
   ```
   to:
   ```
   | Requirement | Phase | Status | Evidence |
   ```

2. **For each of the 159 pending requirements**, update based on evidence classification:

   **DIRECT evidence** (Android test directly exercises the API):
   - Change `[ ]` to `[x]`
   - Set Status to `Complete`
   - Set Evidence to `DIRECT: {test name} passes on Android`

   **INDIRECT evidence** (Android test exercises the code path):
   - Change `[ ]` to `[x]`
   - Set Status to `Complete`
   - Set Evidence to `INDIRECT: {source test/feature} exercises this API on Android`

   **CODE_VERIFIED** (compiles on Android, macOS tests pass, no Android-specific runtime concern):
   - Change `[ ]` to `[x]`
   - Set Status to `Complete`
   - Set Evidence to `CODE_VERIFIED: {macOS test name}; no Android runtime concern`

   **KNOWN_LIMITATION** (cannot work on Android with documented rationale):
   - Keep `[ ]`
   - Set Status to `Known Limitation`
   - Set Evidence to brief rationale (e.g., "Requires SwiftUI DynamicProperty")

   **UNVERIFIED** (insufficient evidence):
   - Keep `[ ]`
   - Set Status to `Pending`
   - Set Evidence to `UNVERIFIED: {reason}`

3. **Update the Coverage summary** at the bottom of the traceability section:
   ```
   - **Complete (evidence-backed):** {N}/184
   - **Known Limitation:** {N}/184
   - **Pending/Unverified:** {N}/184
   ```
