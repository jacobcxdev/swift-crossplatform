# v1.0 Milestone Audit (Revised)

**Milestone:** Swift Cross-Platform TCA on Android via Skip Fuse
**Audited:** 2026-02-24T02:30:00Z
**Status:** **GAPS FOUND — milestone requirements not met**

## Why This Supersedes the Previous Audit

The audit dated 2026-02-23T19:30:00Z claimed "184/184 requirements satisfied" with "status: tech_debt (no critical blockers)". That assessment was based on:
- REQUIREMENTS.md checkbox status (all `[x]`) — but checkboxes were set by phases that only tested on macOS
- Phase VERIFICATION.md files — but these explicitly document that Android testing was deferred or blocked
- Integration checker reports — but these verified code architecture, not runtime behaviour

The previous audit did not flag that **no Android tests have ever successfully run** for the example projects. This revised audit corrects that.

## Executive Summary

The v1.0 milestone defined 184 requirements, nearly all specifying "on Android" as the verification target. All 184 were marked complete. This audit finds that **~144 of those requirements were never tested on Android**. Every phase deferred Android testing to a later phase:

1. Phases 1-6 deferred to Phase 7 (Integration Testing)
2. Phase 7 was blocked by `dlopen`/`dlsym` import errors in xctest-dynamic-overlay
3. Phase 7 documented the blocker as "deferred" and marked itself "passed"
4. Phases 8-10 did documentation/cleanup without resolving the blocker
5. Phase 10 introduced JUnit stubs that make `skip test` report 0 Kotlin tests as "passing"

### What Was Actually Delivered

- **Code architecture**: 19 forks modified with `#if os(Android)` paths. Observation bridge, TCA registrar, navigation adapters designed for cross-platform.
- **macOS test suite**: 257 tests pass on macOS (227 fuse-library + 30 fuse-app).
- **Android compilation**: `make android-build` succeeds for both examples.
- **Documentation**: CLAUDE.md, Makefile, planning docs comprehensive.

### What Was NOT Delivered

- **Android test execution**: No tests have ever run on Android emulator/device for example projects.
- **Canonical Skip testing**: XCSkipTests uses JUnit stubs instead of `XCGradleHarness`/`runGradleTests()`.
- **TCA test transpilation**: TCATests, NavigationTests, FuseAppIntegrationTests lack `skipstone` plugin.
- **swift-perception on Android**: Entirely excluded, gating out Perceptible conformances and binding helpers.
- **Full API parity**: Non-deprecated TCA APIs (`SwitchStore`, `IfLetStore(else:)`, `fullScreenCover`, `popover`) gated out without equivalents.

## Critical Blockers (6)

### 1. ANDROID-TESTING-BYPASS: JUnit Stubs
`XCSkipTests` in both examples creates fake JUnit XML (`tests="0"`) instead of using canonical `XCGradleHarness`/`runGradleTests()`. This makes `make test` / `skip test` report 0 Kotlin tests as passing. Root cause: local fork path overrides break Gradle's skipstone symlink resolution.

### 2. NO-TCA-TESTS-ON-ANDROID: Missing skipstone Plugin
Only 2 of ~8 test targets have the `skipstone` plugin (FuseLibrary, ObservationTests). TCATests, NavigationTests, FoundationTests, SharingTests, DatabaseTests, FuseAppIntegrationTests all lack it — they cannot be transpiled to Kotlin.

### 3. SWIFT-PERCEPTION-EXCLUDED: No Fork for Android
`swift-perception` is entirely gated out on Android (`#if canImport(SwiftUI) && !os(Android)`). This excludes `WithPerceptionTracking`, `_PerceptionLocals`, and `Perceptible` conformances that TCA's binding/scoping infrastructure depends on. The project's "Out of Scope" rationale ("native libswiftObservation.so ships with Android SDK") is incorrect — libswiftObservation provides `@Observable`/`ObservationRegistrar` but NOT the `Perceptible` protocol and perception-tracking debug infrastructure that TCA uses for binding helpers.

### 4. PARITY-GAPS-IN-CURRENT-APIS: Non-Deprecated APIs Missing on Android
These **current, non-deprecated** TCA APIs are `#if !os(Android)` with no equivalent:
- `SwitchStore` / `CaseLet` (enum-driven navigation destinations)
- `IfLetStore(then:else:)` (else branch)
- `ViewActionSending.send(_:animation:)` (animated state sends)
- `fullScreenCover` presentation
- `popover` presentation
- `NavigationStack` programmatic push via path binding (one-directional only)
- Rich `TextState`/`ButtonState` rendering (bold, italic, color silently dropped)

### 5. ANDROID-EMULATOR-NEVER-TESTED: dlopen/dlsym Blocker
Phase 7 verification documents: *"skip android test is blocked by pre-existing dlopen/dlsym import errors in xctest-dynamic-overlay fork (SwiftTesting.swift:643, IsTesting.swift:39 missing #if os(Android) import Android guards)"*. This is a ~4 line fix that was documented but never applied.

### 6. REQUIREMENTS-INTEGRITY: False Completion Claims
All 184 requirements marked `[x]` in REQUIREMENTS.md. ~144 of these specify "on Android" and lack any Android test evidence. The traceability chain is: phase plans claim requirements → verification reads plans → audit reads verification → all say "passed" — but no link in the chain ever ran Android tests.

## Recommendation

1. **Fix blockers in order**: xctest-dynamic-overlay imports → skipstone on test targets → skipstone local package compatibility → swift-perception fork → API parity gaps
2. **Run real Android tests**: Canonical `XCGradleHarness` + `runGradleTests()`, emulator verification via `skip android test`
3. **Re-verify all requirements** against actual Android test results — uncheck everything lacking evidence
4. **Squash and re-init** planning docs before upstream — current commit history claims completion that wasn't achieved
5. **Consider restructuring**: The 10-phase structure may not make sense going forward; a simpler "make it actually work on Android" phase may be more appropriate
